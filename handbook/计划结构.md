# 计划结构

一个主计划文件固定命名为 `plan.json`，它对应一个完整需求，也是最小可执行单元。

同一个需求目录内可以拆分 `sub-plans/*-plan.json` 子计划，供主 `plan.json` 通过 `run_sub_plan` 引用。不同需求目录之间的 `plan.json` 不能互相引用。

需要在单次 plan run 内按间隔触发步骤时，使用 `steps` 中的父级 `trigger` action；周期执行体直接写在 `trigger.steps`，或通过 `trigger.path` 引用当前包内的 `sub-plans/*-plan.json`。不要使用顶层 `routines` 或 `triggers` 定义区。

推荐目录结构：

```text
plan-package/
  plan.json
  config.json
  sub-plans/
    01-open-target-plan.json
    02-fill-form-plan.json
    03-assert-result-plan.json
  resources/
  output/
  profiles/
  docs/
```

## 目录职责

- `plan.json`: 当前需求入口，也是唯一可直接执行的主计划。
- `config.json`: 当前需求的局部配置，只对本 plan 包生效。
- `sub-plans/`: 当前需求内部子计划，只能由本包 `plan.json` 通过 `run_sub_plan` 调用。
- `resources/`: 当前需求独占资源，例如 HTML、CSV、JSON、图片和本地夹具。
- `output/`: 当前需求运行产物，必须由 Git 忽略。
- `profiles/`: 当前需求的本机浏览器持久状态，由 `open_browser.use_profile=true` 自动维护，必须由 Git 忽略。
- `docs/`: 当前需求的说明、前置条件、维护记录和人工验收说明。

## 顶层字段

- `automation_type`: 当前 plan 的执行线，主 `plan.json` 必填。允许值为 `browser` 或 `desktop`
- `variables`: 全局变量字典，供后续步骤通过 `{{变量名}}` 引用
- `steps`: 按顺序执行的动作数组

## 配置文件

- `<plan-root>/config.json`: 当前 plan 集合的共享配置，`plan-root` 来自运行根的 `plan.config.plan_roots`。
- `plan-package/config.json`: 局部 plan 配置，只对当前 plan 包生效。
- 集合配置和局部配置都是 JSON 对象，嵌套对象会递归合并；数组和普通值由局部覆盖全局。
- `config.json` 只保存运行配置，不保存 plan 变量；变量请写在 `plan.json` 的 `variables`。
- 配置字段详见 [config](./reference/config.md)。

## 最小结构

```json
{
  "automation_type": "browser",
  "variables": {},
  "steps": []
}
```

## 执行线类型

`automation_type` 决定执行器如何解析和运行 plan：

- `browser`: 浏览器自动化，使用 Playwright 浏览器会话、页面、DOM selector、网络事件、浏览器截图和浏览器状态。
- `desktop`: 桌面控制，使用 Windows/macOS 桌面 backend、窗口、控件、系统级键鼠、截图、权限诊断和视觉兜底。

主 `plan.json` 必须声明 `automation_type`。`sub-plans/*-plan.json` 默认继承主 plan 的类型；如果子计划显式声明 `automation_type`，必须和主计划一致。一个普通 plan 不能同时混用浏览器和桌面专属 action。

不要把字段命名为顶层 `type`。step 内部已经使用 `type` 表示 action 的具体操作，顶层执行线必须使用 `automation_type`。

## 带变量的常见结构

```json
{
  "automation_type": "browser",
  "variables": {
    "email": "test@example.com",
    "target_url": "{{resources_file_url}}/basic_login.html"
  },
  "steps": [
    {
      "action": "open_browser",
      "name": "main",
      "headed": true
    },
    {
      "action": "navigate",
      "type": "goto",
      "browser": "main",
      "url": "{{target_url}}"
    },
    {
      "action": "element",
      "type": "fill",
      "browser": "main",
      "selector": "input[autocomplete='username']",
      "value": "{{email}}"
    }
  ]
}
```

## 约定

- `automation_type: "browser"` 时，`browser` 字段表示当前步骤作用于哪个浏览器会话，`selector` 使用 Playwright 支持的选择器语法。
- `automation_type: "desktop"` 时，桌面步骤使用 `desktop` 字段表示当前步骤作用于哪个桌面会话；窗口、控件、截图区域、图像和 OCR locator 不使用 Playwright selector 语义。
- `{{email}}` 这种写法会在执行前替换成变量值。
- `{{resources_file_url}}` 是内置变量，指向当前 plan 包的 `resources/`。
- `{{output_dir}}` 是内置变量，指向当前 plan 包的 `output/`。
- 需要把一个步骤的结果交给后续步骤时，使用 step 级 `output` 发布 JSON-safe 输出；后续步骤直接用 `{{变量名.字段}}` 引用，不需要重复声明输入。
- `output.as` 是发布变量名；`output.from` 从当前 action 原始结果中选取片段，支持 `body.data` 或 `/body/data`；`output.type` 和 `output.fields` 是轻量类型校验，例如 `string!`、`object!`、`array`。
- 发布成功后，执行器同时写入 `{{last}}` 指向本次发布值。正式 plan 优先使用命名变量，`last` 只适合紧邻步骤；`last` 是保留输出变量，不能由 `variable` 或 `foreach` 写入。
- plan JSON 内部路径统一使用 `/`，不要使用 Windows 反斜杠；运行时会转换成本机路径。
- 输入资源推荐放在当前 plan 包 `resources/`，例如 CSV、JSON、TXT、HTML、图片、HAR、上传文件、storage state 种子文件和命令 stdin 文件。
- AI 创建 plan 时，如果用户没有指定必须依赖本机固定路径，优先把文件复制或导入到当前包 `resources/`，再在 plan 中引用 `resources/...` 或 `{{resources_file_url}}/...`。
- `C:\...`、`D:\...`、`/Users/...`、`/home/...`、UNC 路径、`file:///C:/...`、其他 plan 包资源或越出当前 plan 包的相对路径都允许作为本机输入路径；工具和校验不因为路径位于 plan 包外而拒绝。
- 所有运行产物必须写到当前 plan 包的 `output/` 下，包括截图、录屏、下载、HTML、JSON、CSV、TXT、storage state、失败截图、失败 HTML 和失败页面状态。
- 浏览器持久登录态只使用当前 plan 包唯一的 `profiles/browser/`，由 `open_browser.use_profile=true` 自动创建和复用；plan 中不写 profile 路径或 profile 名称。
- 输出动作的 `path` 是相对于 `output/` 的路径，不能以 `output/`、`resources/`、`docs/` 或 `sub-plans/` 开头，也不能使用绝对路径。
- 框架按组件固定输出分区，例如截图写入 `output/screenshots/`，下载写入 `output/downloads/`，页面 HTML 写入 `output/html/`，JSON 写入 `output/json/`，CSV 写入 `output/csv/`，文本写入 `output/text/`。
- `read` 读取输入时可以使用 `resources/...`，也可以读取当前 plan 包已经产生的 `output/<component>/...`。
- 参数级别一致的能力统一用 `type` 区分具体操作，例如 `navigate`、`element`、`wait`、`extract`、`assert`、`read`、`write`、`capture`。
- 触发型事件和独立生命周期能力单独保留，例如 `open_browser`、`run_sub_plan`、`trigger`、`wait_for_popup`、`wait_for_download`。
- 浏览器专属 action 不能写进 `desktop` plan，桌面专属 action 不能写进 `browser` plan；跨线 action 应在校验阶段失败。
- 步骤按数组顺序执行，前一步失败时，后续步骤不会继续。

## 推荐写法

- 先 `open_browser`，再 `navigate` + `type: goto`，再做页面交互。
- 在关键节点加 `wait` + `type: selector`，不要盲目依赖固定睡眠。
- 页面值需要复用时，用 `extract` 保存到变量里。
- 人工参与的流程，用 `manual_confirm` 作为暂停点。
- 桌面控制先 `open_desktop`；需要打开 App 时用 `desktop_app type=launch`，再通过窗口列表、等待、聚焦、截图或控件树确认目标后做输入、点击和断言。

## 本机输入路径

```json
{
  "action": "read",
  "type": "csv",
  "path": "D:/business/inbox/source.csv",
  "output": {
    "as": "rows",
    "type": "array!"
  }
}
```

## 步骤输出发布

`output` 用于约束上一个节点“发出去什么”。下游不需要声明输入，只能引用已经发布到变量池的 JSON-safe 值。

```json
{
  "action": "http",
  "type": "request",
  "method": "POST",
  "url": "{{api_base_url}}/login",
  "json": {
    "username": "{{username}}",
    "password": "{{password}}"
  },
  "body_type": "json",
  "output": {
    "as": "login",
    "from": "body.data",
    "type": "object!",
    "fields": {
      "token": "string!",
      "user_id": "string!"
    }
  }
}
```

后续步骤直接引用：

```json
{
  "action": "http",
  "type": "request",
  "method": "GET",
  "url": "{{api_base_url}}/orders",
  "headers": {
    "Authorization": "Bearer {{login.token}}"
  }
}
```

类型声明支持 `string`、`integer`、`number`、`boolean`、`object`、`array`、`null`，末尾加 `!` 表示必须存在且不能为 `null`。
