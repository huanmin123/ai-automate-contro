# 问题-0003-多页面改造后旧引用残留

## 现象

框架已经从单页面模型升级为“一个浏览器会话下支持多个命名页面”，但部分旧逻辑仍然直接访问 `session.page`。

这会导致以下问题：

- `retry` 在失败等待时直接报错
- `sleep` 无法正常等待
- `if` 条件中的 `selector_exists`、`selector_visible` 在多页面版本下失效

## 原因分析

`BrowserSession` 已经改成：

- `pages`
- `current_page_name`
- `require_page()`

但动作层和条件层仍残留单页时代写法，没有完全切换到新接口。

## 修复方案

- `retry` 改为对每个会话的当前活动页执行等待
- `sleep` 改为通过 `require_page()` 获取当前页
- 条件判断改为支持 `page` 字段，并默认落到当前活动页

## 验证结果

- 代码编译通过
- 旧引用已从执行路径移除
- 新增的 popup / request / response / table / csv 示例可以在多页面模型下继续工作

## 预防措施

- 每次运行时模型升级后，补一条错题本记录
- 新增动作时优先复用 `_page()` / `require_page()`，避免绕开统一页面入口
