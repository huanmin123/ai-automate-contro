# 运行说明

## 启动 Textual AI 客户端

```powershell
python .\main.py
```

进入客户端后直接用自然语言描述需求；常用命令只面向 AI 会话和当前工作计划：

```text
/status
/plan
/sessions
/exit
```

交互命令必须写在输入行最开头，格式是 `/command` 或 `/command <args>`。`/` 后面的命令名必须以英文字母开头；命令和参数之间至少一个空格。普通文字中间的 `/xxx` 不会被当作命令，也不会触发命令补全。命令没有重复别名，帮助和候选只展示 AI 会话入口；无 AI 的 plan list/create/validate/run/schedule/debug 使用 `cplan`。

## 创建计划模板

创建空 plan 包属于一次性确定性命令；在 Textual 客户端里直接用自然语言描述需求，让 AI 先探测、再创建 plan。

```powershell
python .\cplan.py create --path .\test-plans\basic\new-plan --automation-type browser
python .\cplan.py create --path .\test-plans\desktop\new-plan --automation-type desktop
```

桌面执行线落地后，`cplan create` 的非交互用法必须显式传入 `--automation-type`。管理端不根据 action 名称猜 plan 类型；`cplan validate` 和 `cplan run` 只读取 `plan.json` 顶层 `automation_type`，再选择浏览器或桌面解析路径。

## 一次性命令

一次性命令用于无 AI 环境、脚本和 CI：

```powershell
python .\cplan.py validate --file .\plans\minimal-browser-plan\plan.json
python .\cplan.py run --file .\plans\minimal-browser-plan\plan.json
python .\cplan.py debug-prepare --file .\test-plans\regression\failure-capture\plan.json --name latest-failure
python .\cplan.py debug-create --file .\test-plans\basic\file-io\plan.json --name file-io-debug
python .\cplan.py list basic
```

## 自检命令

```powershell
python .\cplan.py self-check cli
python .\cplan.py self-check handbook
python .\cplan.py self-check workspace-clean
python .\cplan.py self-check release-matrix
python .\cplan.py self-check browser-components
python .\cplan.py self-check desktop-env
python .\cplan.py self-check desktop-examples
python .\cplan.py self-check desktop-components
python .\cplan.py self-check desktop-real-app
python .\main.py self-check ai-desktop-loop
python .\main.py self-check ai-real-execution-line --api-key-file D:\模型密钥.txt
```

`desktop-env` 只检查桌面控制环境，不移动鼠标、不启动真实 App。默认是诊断模式；发布前可按需加 `--require-input`、`--require-vision`、`--require-ocr`、`--require-ocr-zh` 强制 native backend probe、Windows PowerShell 7 和依赖可用。

`desktop-real-app` 会动态创建临时 plan 包，按当前运行环境选择可用系统 App 和临时窗口夹具；在 Windows 上覆盖 Notepad、Explorer、可见 PowerShell 终端和临时 WinForms Open/Save 文件对话框，在 macOS 上覆盖 TextEdit 或系统可用轻量 App。终端回归会验证窗口启动、窗口查询、活动窗口读取、截图、剪贴板输入命令、回车执行、结果文件断言和退出清理。无 GUI、锁屏、权限不足或依赖缺失时返回 `skipped` 和原因。

`release-matrix` 默认运行当前机器可执行的确定性提交前矩阵，不调用真实模型。需要包含真实 AI 回归时显式传入：

```powershell
python .\cplan.py self-check release-matrix --include-real-ai --api-key-file D:\模型密钥.txt
```

定位单项问题时可以先列出步骤，再只运行指定项：

```powershell
python .\cplan.py self-check release-matrix --list
python .\cplan.py self-check release-matrix --only handbook --only workspace_clean --fail-fast
python .\cplan.py self-check release-matrix --only desktop_env,desktop_examples,desktop_components,desktop_real_app --step-timeout-seconds 1200
python .\cplan.py self-check release-matrix --strict-desktop --fail-fast --step-timeout-seconds 1200
python .\cplan.py self-check release-matrix --only desktop_env,desktop_examples,desktop_components,desktop_real_app,ai_desktop_loop --repeat 3 --fail-fast --step-timeout-seconds 1200
```

`--strict-desktop` 等价于同时启用 `--require-desktop-input --require-desktop-vision --require-desktop-ocr --require-desktop-ocr-zh`，用于当前机器发布前强制桌面输入、视觉和 OCR 真实可用。`--repeat` 会按矩阵顺序重复执行选中的步骤，并在结果里记录 `iteration/repeat`。它用于定位桌面焦点、系统窗口、终端窗口、文件对话框、视觉截图和 OCR 这类偶发问题；真实 AI 回归仍需显式加 `--include-real-ai`。

## 长期定时计划

`cplan schedule` 用于按时间启动完整 plan run。它是 CLI 管理层能力，不写入普通 `plan.json.steps`。

```powershell
python .\cplan.py schedule list
python .\cplan.py schedule add --id vsllm_daily --file D:\my-plans\vsllm-daily-checkin-card\plan.json --daily-at 09:05 --project-root D:\my-plans
python .\cplan.py schedule add --id poll_demo --file .\test-plans\control-flow\interval-trigger\plan.json --every-seconds 300 --run-immediately
python .\cplan.py schedule run-now vsllm_daily
python .\cplan.py schedule daemon --once
python .\cplan.py schedule daemon
python .\cplan.py schedule disable vsllm_daily
python .\cplan.py schedule remove vsllm_daily
```

配置写入当前运行根的 `schedules.json`；运行状态写入 `.keygen/schedules-state.json`。删除最后一个 schedule 后，空配置和空状态文件会自动移除。`daemon` 是常驻扫描进程，`--once` 只扫描一次后退出，适合测试或交给 Windows 任务计划程序定时调用。

## 持续 AI 终端

AI 终端是 plan 级助手，不是一次性命令。它会在当前终端会话中保留上下文，适合让 AI 帮你读取 plan、运行验证、分析输出、创建调试工作区和生成补丁。

AI 创建 plan 前必须先判断执行线。用户明确描述网页、URL、浏览器、DOM 或网页后台时，创建 `automation_type: "browser"` plan；用户明确描述本机桌面应用、窗口、系统键鼠、Windows/macOS 应用或授权时，创建 `automation_type: "desktop"` plan。描述不明确时，AI 终端必须先追问确认，不能直接写 plan。

桌面线按当前运行环境判断平台和可用能力。`Notepad/Explorer/PowerShell/客户端窗口/Open/Save 文件对话框/系统键鼠` 这类本机桌面对象属于 desktop；但 `Chrome/浏览器/网页/URL` 仍属于 browser，即使用户同时提到 Windows 或 macOS 也不能误判为桌面。desktop plan 写入和运行前要先通过 `inspect_desktop` 获取 `capability_matrix`、窗口列表、权限/依赖、控件或截图探测证据；plan 内优先用 `desktop_capture type=observe` 保留统一观察证据，并按需要补充窗口、控件、截图、等待或断言等运行证据；缺少这些证据时 `review_plan_quality` 会失败。Open/Save 文件对话框按真实桌面窗口处理，推荐 `desktop_wait window -> desktop_capture screenshot -> desktop_input type_text method=clipboard -> desktop_input hotkey enter -> desktop_wait not_exists`。

启动 AI 终端前必须能在 PowerShell 7 中运行 `rg --version`。AI 终端的文本定位只支持跨平台 `ripgrep`，不会退回 Windows 内置搜索；缺失时先执行 `winget install --id BurntSushi.ripgrep.MSVC -e`，或让用户确认后由助手帮助全局安装。

```powershell
python .\main.py ai
python .\main.py ai --thread login-debug
python .\main.py ai --thread login-debug ask --message "读取当前 plan 并给出下一步" --json
```

进入后可以直接对话：

```text
请通过工具列出 ai 相关 plan，并告诉我有几个。
读取 test-plans\ai\controlled-text\plan.json，解释这个 plan 做什么。
```

AI 终端使用 `test-plans/config.json` 的 `ai_services.default`。这个配置里保留临时真实测试服务的 base_url、model 和调用参数，用于 AI 场景回归；真实密钥可以通过 `api_key` 直接写入，也可以通过 `api_key_env` 指向本机环境变量，项目不做自动脱敏或拒写。

`ai ask` 是非交互回归入口，适合脚本里验证真实模型能否完成一次工具调用或 plan 创建任务。它不会引入另一套协议，仍然使用 LangChain/OpenAI-compatible 原生消息、tool_calls、checkpoint 和压缩链路；如果模型服务返回 503、欠费、协议不兼容、schema 不符合或 LangChain 抛出返回对象错误，直接把错误暴露给用户。调试真实网站流程时可加 `--events`，程序会用 JSONL 实时输出工具调用、plan 运行进度、人工确认和错误事件。`ai ask` 可以验证流程会正确停在 `manual_confirm`，但它不会等待人工操作，也不能在进程结束后保留可继续接管的 Playwright 浏览器；需要人工登录、验证码、二次验证或菜单操作的流程应在 Textual 活跃客户端里发起，并在同一个客户端和同一个可见 Playwright 浏览器窗口中回复“继续”。

AI 终端会把 LangGraph checkpoint 写入本地 `.keygen/ai-terminal-checkpoints.sqlite`，并把会话列表摘要维护到 `.keygen/ai-terminal-sessions/index.json`。同一个 `--thread` 可以跨终端进程恢复上下文；进入客户端后可用 `/status` 查看统一状态，`/sessions [limit|all]` 查询已保存会话摘要，`/resume <thread-id-or-index>` 选择会话继续沟通，`/new [thread-id]` 开新线程。

常用 AI 终端会话命令：

```text
/status
/sessions all
/sessions 20
/resume login-debug
/new
```

AI 终端请求由后台 worker 执行。发送自然语言后，Textual 客户端会用灰色输入块回显用户消息，AI 回复直接在普通背景输出，不再带发言人前缀；当前输入行是灰底无边框多行 composer，内容换行或窗口变窄后高度会在 4 到 8 行内增长。AI 调用工具时会即时显示一行工具进度，并在工具完成后原地更新为完成或失败状态，避免刷屏；系统输出、审批和错误分别显示为独立块，不再混在工具进度里。AI 正在回复或运行工具时继续发送的普通消息会进入队列，底部状态持续展示服务、thread、AI 上下文中的当前 plan、debug workspace、最近 output、排队数量、审批状态和待发送图片。模型服务报错会直接显示 service、model、base_url 和上游错误。AI 返回的 Markdown 只由 Textual 客户端展示层渲染；原始 Markdown 仍然写入 LangGraph checkpoint、压缩归档和会话索引，模型请求也不包含任何终端渲染字段。需要留存当前屏幕对话时，用 `/export [path]` 导出 Markdown；只保存最近一条 AI 回复时，用 `/copy-last [path]`；清空当前屏幕用 `/clear`。需要排查模型连通性时使用一次性命令 `python .\main.py ai check --json`。

常用状态命令：

```text
/status
/sessions 20
/resume 1
```

AI 终端的当前 plan、调试工作区和最近输出目录会在选择、运行和调试 plan 时自动更新，并由 LangGraph checkpoint 保存。更新后可以直接说“读取当前 plan”“分析最近输出”“运行当前 debug workspace”，这些上下文会在模型调用前自动注入。

会话压缩使用 LangChain `SummarizationMiddleware`。项目按 128k token 上下文标准估算，达到 50% 时自动压缩，也就是约 64k tokens 触发，压缩后保留约 32k tokens 的近期上下文。每次压缩都会把完整消息归档到当前线程目录：

```text
.keygen/ai-terminal-sessions/<thread>/compressions/<timestamp>-<reason>/
  messages.jsonl
  summary.md
  manifest.json
```

模型现场只保留压缩摘要、最近消息和归档路径；会话索引只保存线程、时间、消息计数、图片计数、最近消息预览和上下文路径摘要，不保存完整对话。需要历史细节时先读 `summary.md`，再按需用 `grep_project_text` 和 `read_project_file_slice` 读取 `messages.jsonl` 的小范围片段。压缩摘要生成如果遇到模型服务 503、欠费、协议不兼容或其他调用错误，会直接报错给用户，不做降级或兼容兜底。

AI 终端支持图片作为当前用户消息的一部分。主路径是截图或复制图片后，在输入行按 `Ctrl+V`；如果剪贴板是图片或图片文件，客户端会插入 `[图片 #n]` 图片标记，用户继续输入文字即可。已经保存成文件的图片可用 `/image <image-path>` 作为兜底入口。输入行支持 `/` 命令提示：输入 `/` 展开候选，继续输入前缀会过滤候选，按 `Up` / `Down` 选择，按 `Tab` 或 `Enter` 补全命令；带参数的命令继续按 `Enter` 发送。附件会复制到当前线程的 `.keygen/ai-terminal-sessions/<thread>/attachments/`，发送给模型前转换为 data URL；发送成功后清空，模型服务报错时保留，方便用户重试。

AI 终端输入行支持多行消息。普通 `Enter` 发送；`Shift+Enter`、`Alt+Enter` 或行尾输入 `\` 后按 `Enter` 会插入换行；多行粘贴保留换行。单独按 `Esc` 中断当前 AI 回复或 plan 运行，已有排队消息时会把队列内容作为下一轮介入消息优先处理。

模型请求只使用 OpenAI-compatible 结构。会话 thread、上下文摘要路径、附件 metadata 和本地事件不会作为自定义字段发给模型；上下文以普通 system message 文本注入，图片以标准 `content` 列表发送，其中图片块为 `{"type": "image_url", "image_url": {"url": "data:image/..."}}`。

```text
[图片 #1] 看一下这个页面为什么没有出现提交按钮
```

图片文件路径仍可作为兜底入口：

```text
/image .\docs\screenshots\login-error.png
```

交互客户端依赖 Textual，Markdown 展示层依赖 Rich，图片附件处理依赖 Pillow；正常安装项目依赖即可：

```powershell
python -m pip install -e .
```

当 AI 准备应用 debug patch 时，Textual 客户端会暂停当前工具调用并显示独立审批块。确认 patch 后输入：

```text
/approve
```

拒绝时输入：

```text
/reject 这里写拒绝原因
```

## 结构化工具入口

AI 终端底层调用同一组确定性工具。脚本和回归可以直接调用这些工具，输出固定为 JSON：

```powershell
python .\main.py tool list
python .\main.py tool check
python .\main.py self-check env
python .\main.py self-check textual-client
python .\main.py self-check ai-stream
python .\main.py self-check ai-terminal
python .\main.py self-check ai-tools
python .\main.py tool schema validate_plan
python .\main.py tool call list_plan_packages --args-json '{"filter_text":"ai"}'
python .\main.py tool call validate_plan --args-json '{"plan_path":".\test-plans\ai\controlled-text\plan.json"}'
python .\main.py tool call read_latest_run_state --args-json '{"plan_path":".\test-plans\ai\controlled-text\plan.json"}'
python .\main.py tool call read_latest_run_report --args-json '{"plan_path":".\test-plans\ai\controlled-text\plan.json"}'
python .\main.py tool call analyze_latest_run_failure --args-json '{"plan_path":".\test-plans\regression\failure-capture\plan.json"}'
python .\main.py tool call read_debug_workspace --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>"}'
python .\main.py tool call list_schedules --args-json '{}'
python .\main.py tool call add_schedule --args-json '{"schedule_id":"demo","plan_path":".\test-plans\control-flow\interval-trigger\plan.json","every_seconds":300,"run_immediately":true}'
```

结构化工具参数由共享 Pydantic schema 校验，AI 终端、LangChain `StructuredTool` 和 `python .\main.py tool call` 使用同一套参数规则。`python .\main.py tool check` 会检查工具函数注册表、参数 schema 和描述是否完全对齐；`python .\main.py self-check ai-tools` 会真实构建 LangChain `StructuredTool`、调用只读工具，验证渐进式读取工具不会返回整包 JSON 正文，并验证受保护工具必须处于 HITL approve resume 才能执行。新增工具后这两个命令都必须运行。AI 新建 plan 包时可以用 `create_plan_package` 和 `write_plan_package_file` 写入受控白名单文件：`plan.json`、`config.json`、`docs/**`、`resources/**`、`sub-plans/*-plan.json`；用户提供本机输入文件但没有明确要求长期依赖该路径时，用 `import_plan_resource_file` 复制到当前 plan 包 `resources/` 后再写 plan；这些工具只拒绝 `output/`、`.keygen/`、缓存、pyc、egg-info 等非 plan 包结构路径，不因账号、密码、token 或 api_key 等明文内容拒写。plan action 的运行证据和中间产物仍写入当前 plan 包 `output/`；用户明确要求最终文件保存到 `Downloads`、桌面或绝对路径时，AI 终端应在产物生成后调用 `export_local_file` 直接写入或从 `output/` 复制到目标路径，不要求用户手动复制。AI 为真实网站创建最终 plan 前必须先用 `inspect_web_page` 真实访问页面并读取受限 DOM 摘要；涉及登录、验证码、后台菜单或动态页面时，应继续创建并运行 `open_browser.headed=true` 的探索 plan，把流程先跑通。需要用户介入时，必须在同一个 Playwright 浏览器窗口里用 `manual_confirm` 交接，不能让用户另开本机浏览器后再提供 URL/截图来替代自动化交接。涉及已有原始 plan 的修复仍必须走 debug workspace、补丁和用户确认。AI 修改 debug workspace 内 JSON plan/config 时优先用 `patch_debug_workspace_json` 做路径级最小修改；需要整文件写入、文档、资源、notes 或 report 时再用 `write_debug_workspace_file`。AI 终端调用 `apply_debug_patch_after_approval` 时会由 LangChain `HumanInTheLoopMiddleware` 中断等待用户 `/approve` 或 `/reject`。

新增或变更 AI 终端工具时，按 [AI 终端工具开发检查清单](./AI终端工具开发检查清单.md) 同步工具函数、Pydantic schema、描述、CLI 回归和真实 AI 回归。

专项 AI 默认使用 `response_format: "json_schema"`，也可以显式配置为 `"json_object"` 或 `"plain"`。用户提供的模型服务如果欠费、503、协议不兼容、返回空内容或不符合 schema，会直接失败，不做自动降级、手动重试或兼容兜底。SDK/LangChain 自身的传输重试可以通过 `max_retries` 显式配置。

专项 AI 使用 `chat_completions` 时支持配置 `"stream": true`。开启 streaming 后框架会从 chunks 中还原文本，再执行 JSON 解析和 schema 校验。

AI 终端的线程上下文只保存当前 plan、当前 debug workspace、最近输出目录、最近压缩摘要路径和归档路径。模型调用前注入的是这些路径摘要，不会把完整 `run.log`、`events.jsonl`、`commands.jsonl` 或大型产物塞进 system prompt。文本读取采用渐进式流程：先用 `read_plan_package`、`read_debug_workspace`、`list_output_artifacts` 获取结构、路径、大小和 step outline，不返回完整 plan/config/sub-plan 正文；再用 `grep_project_text` 通过 `rg` 定位关键词，最后用 `read_project_file_slice` 或小范围 artifact 读取拿必要行段。需要运行证据时优先使用 `read_latest_run_report`、`read_latest_run_state`、`read_run_log --lines <n>`、`read_run_events --lines <n>` 和 `analyze_latest_run_failure` 读取尾部摘要。

`python .\main.py self-check ai-terminal` 是本地确定性回归，不访问真实模型，用于检查 AI 终端会话摘要、`resume` 选择器、常用命令流、压缩归档、图片附件 metadata 和图片 data URL 原文保留。

`python .\main.py ai check --json` 会发送一次真实模型请求，复用 AI 终端的配置、LangGraph agent 和 checkpoint 链路；它适合排查 `self-check env` 通过但交互里仍报错的情况。`self-check env` 只证明本地配置结构和 key 存在，不能证明账号额度、模型名、网关状态和 OpenAI-compatible 协议此刻可用。

`python .\main.py self-check ai-stream` 是本地确定性回归，不访问真实模型，用于检查 chat completions streaming chunk、reasoning chunk 忽略、SDK 对象 chunk 和空流拒绝逻辑。修改 `src/ai_automate_contro/ai/response_parsing.py` 的 streaming 解析时必须运行。

## 输出目录

- 每次运行都会写入对应 plan 包的 `output/`
- 每次运行目录包含 `state.json`、`result.json`、`report.md`、`run.log`、`events.jsonl`，交互式运行还会包含 `commands.jsonl`
- 动作组件产物按类型写入固定分区，例如 `output/screenshots/`、`output/json/`、`output/variables/`、`output/ai/`
- `state.json` 是 Textual 客户端、AI 后端和外部脚本读取当前 run 状态的稳定入口
- `output/` 属于运行产物，不应提交

## 人工确认流程

`manual_confirm` 会让当前 run 进入等待态。无 AI 环境使用 `cplan run` 时，在当前命令行只输入 `y` 或 `n`：`y` 表示继续/确认，`n` 表示停止/拒绝；其他输入会重新提示：

```powershell
python .\cplan.py run --file .\test-plans\basic\manual-confirm\plan.json
```

AI 通过工具运行到确认点时，确认提示会回到当前 Textual 对话；用户用自然语言回复继续、停止或补充问题即可。Textual 客户端没有 `/continue`、`/stop`、`/run`、`/validate` 这类 plan 管理命令。

## 调试工作区

AI 终端和人工排查都可以用调试工作区隔离原始 plan。Textual 客户端不再提供 `/debug`、`/artifacts` 等低频调试命令；日常使用优先直接让 AI 创建、运行和修复 plan。人工排查用一次性 CLI：

```powershell
python .\cplan.py debug-prepare --file .\test-plans\regression\failure-capture\plan.json --name latest-failure
python .\cplan.py debug-create --file .\test-plans\basic\file-io\plan.json --name file-io-investigation
python .\cplan.py debug-inject --workspace .\test-plans\basic\file-io\output\debug\<run> --preset print --preset variables
python .\cplan.py debug-patch --workspace .\test-plans\basic\file-io\output\debug\<run>
python .\main.py tool call list_output_artifacts --args-json '{"plan_path":".\test-plans\basic\file-io\plan.json","filter_text":"debug","limit":20}'
```

工作区创建在当前 plan 包：

```text
output/debug/<timestamp>-<name>/
  source-copy/
  injected-plan/
  notes.md
  report.md
  patch.diff
  manifest.json
```

`source-copy/` 是原始 plan 快照；`injected-plan/` 用于注入调试用的 `print`、截图、HTML、变量快照和 `manual_confirm`。

`debug prepare [name]` 会读取最近失败运行的截图、HTML、页面状态和 DOM 摘要，创建 debug workspace，并把诊断步骤插到失败 step 之前。这样调试副本可以在失败动作执行前再次保存页面状态，避免把诊断步骤追加到永远执行不到的末尾。

支持的注入 preset：

- `print`: 在运行日志中写入调试检查点。
- `variables`: 写出变量快照到 `output/variables/debug/`。
- `manual_confirm`: 在当前可见 Playwright 浏览器窗口中等待用户完成登录、验证码、二次验证或页面检查。
- `screenshot`: 截图，需要指定浏览器会话名。
- `html`: 保存页面 HTML，需要指定浏览器会话名。

一次性注入命令：

```powershell
python .\cplan.py debug-inject --workspace .\test-plans\basic\file-io\output\debug\<run> --preset print --preset variables --message "debug checkpoint"
python .\cplan.py debug-inject --workspace .\test-plans\basic\file-io\output\debug\<run> --preset screenshot --preset html --browser demo --position before_step --step 3
python .\cplan.py debug-patch --workspace .\test-plans\basic\file-io\output\debug\<run>
```

运行调试副本：

```powershell
python .\cplan.py run --file .\test-plans\basic\file-io\output\debug\<run>\injected-plan\plan.json --run-name injected-debug
```

使用结构化工具完成调试副本修复闭环：

```powershell
python .\main.py tool call read_debug_workspace --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>"}'
python .\main.py tool call propose_debug_fix --args-json '{"workspace":".\test-plans\regression\failure-capture\output\debug\<run>","user_hint":"系统账户"}'
python .\main.py tool call propose_debug_fix --args-json '{"workspace":".\test-plans\regression\failure-capture\output\debug\<run>","user_hint":"系统账户","apply":true,"run_after_apply":true}'
python .\main.py tool call patch_debug_workspace_json --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>","root":"injected-plan","relative_path":"plan.json","operations":[{"op":"replace","path":["steps",0,"message"],"value":"debug message"}]}'
python .\main.py tool call write_debug_workspace_file --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>","root":"injected-plan","relative_path":"plan.json","content":"..."}'
python .\main.py tool call validate_debug_plan --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>"}'
python .\main.py tool call run_debug_plan --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>","run_name":"injected-debug"}'
python .\main.py tool call generate_debug_patch --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>"}'
```

确认修复后应用补丁：

```powershell
python .\cplan.py debug-apply --workspace .\test-plans\basic\file-io\output\debug\<run> --yes
```

不带 `--yes` 不会修改原始 plan 包。

`propose_debug_fix` 当前只做保守的候选修复，第一版覆盖 selector 超时：从失败 HTML 的 DOM 摘要中提取候选 selector，并结合用户 hint 排序。它应用候选时会先把 `source-copy/plan.json` 复制回 `injected-plan/plan.json`，再写入最小修复，避免调试注入污染最终 patch。
