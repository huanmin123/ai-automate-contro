# 运行说明

## 启动持续管理终端

```powershell
python .\main.py
```

进入终端后运行参考计划：

```powershell
/use .\plans\minimal-browser-plan\plan.json
/inspect
/validate
/run
/logs 20
/events 20
/report
/artifacts
/debug create demo-investigation
/debug inject print,variables
/debug patch
/status
/status --short
/output
/quit
```

交互命令必须写在输入行最开头，格式是 `/command` 或 `/command <args>`。`/` 后面的命令名必须以英文字母开头；命令和参数之间至少一个空格。普通文字中间的 `/xxx` 不会被当作命令，也不会触发命令补全。

## 创建计划模板

在持续管理终端中：

```powershell
/create .\test-plans\basic\new-plan
```

或者使用一次性命令：

```powershell
python .\main.py plan create --path .\test-plans\basic\new-plan
```

## 一次性命令

一次性命令用于脚本、CI 和后续 AI 工具调用：

```powershell
python .\main.py plan validate --file .\plans\minimal-browser-plan\plan.json
python .\main.py plan run --file .\plans\minimal-browser-plan\plan.json
python .\main.py plan debug-prepare --file .\test-plans\regression\failure-capture\plan.json --name latest-failure
python .\main.py plan debug-create --file .\test-plans\basic\file-io\plan.json --name file-io-debug
python .\main.py plan list basic
```

## 持续 AI 终端

AI 终端是 plan 级助手，不是一次性命令。它会在当前终端会话中保留上下文，适合让 AI 帮你读取 plan、运行验证、分析输出、创建调试工作区和生成补丁。

启动 AI 终端前必须能在 PowerShell 7 中运行 `rg --version`。AI 终端的文本定位只支持跨平台 `ripgrep`，不会退回 Windows 内置搜索；缺失时先执行 `winget install --id BurntSushi.ripgrep.MSVC -e`，或让用户确认后由助手帮助全局安装。

```powershell
python .\main.py ai
python .\main.py ai --thread login-debug
python .\main.py ai --thread login-debug ask --message "读取当前 plan 并给出下一步" --json
```

进入后可以直接对话：

```text
请通过工具列出 ai 相关 plan，并告诉我有几个。
读取 test-plans\ai\controlled-text\plan.json，解释这个 plan 做什么。
```

AI 终端使用 `test-plans/config.json` 的 `ai_services.default`。这个配置里当前包含用户提供的临时真实测试服务，用于 AI 场景回归，不要在清理代码或文档时删除。

`ai ask` 是非交互回归入口，适合脚本里验证真实模型能否完成一次工具调用或 plan 创建任务。它不会引入另一套协议，仍然使用 LangChain/OpenAI-compatible 原生消息、tool_calls、checkpoint 和压缩链路；如果模型服务返回 503、欠费、协议不兼容、schema 不符合或 LangChain 抛出返回对象错误，直接把错误暴露给用户。

AI 终端会把 LangGraph checkpoint 写入本地 `.keygen/ai-terminal-checkpoints.sqlite`，并把会话列表摘要维护到 `.keygen/ai-terminal-sessions/index.json`。同一个 `--thread` 可以跨终端进程恢复上下文；进入终端后可用 `/status` 查看当前状态，`/sessions [limit|all]` 查询已保存会话摘要，`/resume <thread-id-or-index>` 选择会话继续沟通，`/history [limit]` 查看近期消息，`/new [thread-id]` 开新线程。

常用 AI 终端会话命令：

```text
/status
/sessions all
/sessions 20
/resume login-debug
/new
/compress handoff-before-long-debug
/compact handoff-before-long-debug
```

AI 终端请求由后台 worker 执行。发送自然语言后，终端会用灰色输入块回显用户消息，AI 回复直接在普通背景输出，不再带发言人前缀；当前输入行是灰底无边框多行 composer，内容换行或窗口变窄后高度会在 1 到 6 行内增长。AI 调用工具时会即时显示工具进度块，包含工具名、关键参数、完成/失败状态和简短结果摘要，避免长时间空白等待；需要完整细节时输入 `/details` 切换显示。AI 正在回复或运行工具时继续发送的普通消息会进入队列，底部状态展示正在处理和排队数量。模型服务报错会直接显示。默认展示层会渲染 AI 返回的 Markdown，支持标题、加粗、列表和代码块；原始 Markdown 仍然写入 LangGraph checkpoint、压缩归档和会话索引，模型请求也不包含任何终端渲染字段。可用 `/render plain` 切换到原始文本显示，用 `/render markdown` 恢复。需要留存当前屏幕对话时，用 `/export [path]` 导出 Markdown。

常用状态命令：

```text
/status
/sessions 20
/resume 1
/compress handoff-before-long-debug
```

AI 终端的当前 plan、调试工作区和最近输出目录会在选择、运行和调试 plan 时自动更新，并由 LangGraph checkpoint 保存。更新后可以直接说“读取当前 plan”“分析最近输出”“运行当前 debug workspace”，这些上下文会在模型调用前自动注入。

会话压缩使用 LangChain `SummarizationMiddleware`。项目按 128k token 上下文标准估算，达到 50% 时自动压缩，也就是约 64k tokens 触发，压缩后保留约 32k tokens 的近期上下文。用户也可以手动输入 `/compress [reason]`。每次压缩都会把完整消息归档到当前线程目录：

```text
.keygen/ai-terminal-sessions/<thread>/compressions/<timestamp>-<reason>/
  messages.jsonl
  summary.md
  manifest.json
```

模型现场只保留压缩摘要、最近消息和归档路径；会话索引只保存线程、时间、消息计数、图片计数、最近消息预览和上下文路径摘要，不保存完整对话。需要历史细节时先读 `summary.md`，再按需用 `grep_project_text` 和 `read_project_file_slice` 读取 `messages.jsonl` 的小范围片段。压缩摘要生成如果遇到模型服务 503、欠费、协议不兼容或其他调用错误，会直接报错给用户，不做降级或兼容兜底。

AI 终端支持图片作为当前用户消息的一部分。主路径是截图或复制图片后，在输入行按 `Ctrl+V`；如果剪贴板是图片或图片文件，客户端会插入 `[图片 #n]` 图片标记，用户继续输入文字即可。已经保存成文件的图片可用 `/image <image-path>` 作为兜底入口。输入行支持 `/` 命令提示：输入 `/` 展开候选，继续输入前缀会过滤候选，按 `Up` / `Down` 选择，按 `Tab` 或 `Enter` 补全命令；带参数的命令继续按 `Enter` 发送。附件会复制到当前线程的 `.keygen/ai-terminal-sessions/<thread>/attachments/`，发送给模型前转换为 data URL；发送成功后清空，模型服务报错时保留，方便用户重试。

AI 终端输入行支持多行消息。`Ctrl+J` 插入换行，普通 `Enter` 始终发送消息。

模型请求只使用 OpenAI-compatible 结构。会话 thread、上下文摘要路径、附件 metadata 和本地事件不会作为自定义字段发给模型；上下文以普通 system message 文本注入，图片以标准 `content` 列表发送，其中图片块为 `{"type": "image_url", "image_url": {"url": "data:image/..."}}`。

```text
[图片 #1] 看一下这个页面为什么没有出现提交按钮
```

图片文件路径仍可作为兜底入口：

```text
/image .\docs\screenshots\login-error.png
```

交互客户端依赖 Textual，Markdown 展示层依赖 Rich，图片附件处理依赖 Pillow；正常安装项目依赖即可：

```powershell
python -m pip install -e .
```

当 AI 准备应用 debug patch 时，终端会暂停并显示 `[WAIT_APPROVAL]`。确认 patch 后输入：

```text
/approve
```

拒绝时输入：

```text
/reject 这里写拒绝原因
```

## 结构化工具入口

AI 终端底层调用同一组确定性工具。脚本和回归可以直接调用这些工具，输出固定为 JSON：

```powershell
python .\main.py tool list
python .\main.py tool check
python .\main.py self-check env
python .\main.py self-check textual-client
python .\main.py self-check ai-stream
python .\main.py self-check ai-terminal
python .\main.py self-check ai-tools
python .\main.py tool schema validate_plan
python .\main.py tool call list_plan_packages --args-json '{"filter_text":"ai"}'
python .\main.py tool call validate_plan --args-json '{"plan_path":".\test-plans\ai\controlled-text\plan.json"}'
python .\main.py tool call read_latest_run_state --args-json '{"plan_path":".\test-plans\ai\controlled-text\plan.json"}'
python .\main.py tool call read_latest_run_report --args-json '{"plan_path":".\test-plans\ai\controlled-text\plan.json"}'
python .\main.py tool call analyze_latest_run_failure --args-json '{"plan_path":".\test-plans\regression\failure-capture\plan.json"}'
python .\main.py tool call read_debug_workspace --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>"}'
```

结构化工具参数由共享 Pydantic schema 校验，AI 终端、LangChain `StructuredTool` 和 `python .\main.py tool call` 使用同一套参数规则。`python .\main.py tool check` 会检查工具函数注册表、参数 schema 和描述是否完全对齐；`python .\main.py self-check ai-tools` 会真实构建 LangChain `StructuredTool`、调用只读工具，验证渐进式读取工具不会返回整包 JSON 正文，并验证受保护工具必须处于 HITL approve resume 才能执行。新增工具后这两个命令都必须运行。AI 新建 plan 包时可以用 `create_plan_package` 和 `write_plan_package_file` 写入受控白名单文件：`plan.json`、`config.json`、`docs/**`、`resources/**`、`sub-plans/*-plan.json`；这个工具拒绝 `output/`、`.keygen/`、缓存、pyc 和 egg-info 路径。AI 为真实网站创建最终 plan 前必须先用 `inspect_web_page` 真实访问页面并读取受限 DOM 摘要；涉及登录、验证码、后台菜单或动态页面时，应继续创建并运行 `open_browser.headed=true` 的探索 plan，把流程先跑通。需要用户介入时，必须在同一个 Playwright 浏览器窗口里用 `manual_confirm` 交接，不能让用户另开本机浏览器后再提供 URL/截图来替代自动化交接。涉及已有原始 plan 的修复仍必须走 debug workspace、补丁和用户确认。AI 修改 debug workspace 内 JSON plan/config 时优先用 `patch_debug_workspace_json` 做路径级最小修改；需要整文件写入、文档、资源、notes 或 report 时再用 `write_debug_workspace_file`。AI 终端调用 `apply_debug_patch_after_approval` 时会由 LangChain `HumanInTheLoopMiddleware` 中断等待用户 `/approve` 或 `/reject`。

新增或变更 AI 终端工具时，按 [AI 终端工具开发检查清单](./AI终端工具开发检查清单.md) 同步工具函数、Pydantic schema、描述、CLI 回归和真实 AI 回归。

专项 AI 默认使用 `response_format: "json_schema"`，也可以显式配置为 `"json_object"` 或 `"plain"`。用户提供的模型服务如果欠费、503、协议不兼容、返回空内容或不符合 schema，会直接失败，不做自动降级、手动重试或兼容兜底。SDK/LangChain 自身的传输重试可以通过 `max_retries` 显式配置。

专项 AI 使用 `chat_completions` 时支持配置 `"stream": true`。开启 streaming 后框架会从 chunks 中还原文本，再执行 JSON 解析和 schema 校验。

AI 终端的线程上下文只保存当前 plan、当前 debug workspace、最近输出目录、最近压缩摘要路径和归档路径。模型调用前注入的是这些路径摘要，不会把完整 `run.log`、`events.jsonl`、`commands.jsonl` 或大型产物塞进 system prompt。文本读取采用渐进式流程：先用 `read_plan_package`、`read_debug_workspace`、`list_output_artifacts` 获取结构、路径、大小和 step outline，不返回完整 plan/config/sub-plan 正文；再用 `grep_project_text` 通过 `rg` 定位关键词，最后用 `read_project_file_slice` 或小范围 artifact 读取拿必要行段。需要运行证据时优先使用 `read_latest_run_report`、`read_latest_run_state`、`read_run_log --lines <n>`、`read_run_events --lines <n>` 和 `analyze_latest_run_failure` 读取尾部摘要。

`python .\main.py self-check ai-terminal` 是本地确定性回归，不访问真实模型，用于检查 AI 终端会话摘要、`resume` 选择器、常用命令流、压缩归档、图片附件 metadata 和 base64 redaction。

`python .\main.py self-check ai-stream` 是本地确定性回归，不访问真实模型，用于检查 chat completions streaming chunk、reasoning chunk 忽略、SDK 对象 chunk 和空流拒绝逻辑。修改 `src/ai_automate_contro/ai/response_parsing.py` 的 streaming 解析时必须运行。

## 输出目录

- 每次运行都会写入对应 plan 包的 `output/`
- 每次运行目录包含 `state.json`、`result.json`、`report.md`、`run.log`、`events.jsonl`，交互式运行还会包含 `commands.jsonl`
- 动作组件产物按类型写入固定分区，例如 `output/screenshots/`、`output/json/`、`output/variables/`、`output/ai/`
- `state.json` 是管理终端、后续 AI 终端和外部脚本读取当前 run 状态的稳定入口
- `output/` 属于运行产物，不应提交

## 人工确认流程

`manual_confirm` 会让当前 run 进入等待态：

```powershell
/use .\test-plans\basic\manual-confirm\plan.json
/run
/status
/continue
/status
```

等待时可以执行：

- `/continue`: 继续当前 run
- `/stop`: 终止等待中的 run
- `/status`: 查看当前等待信息或最近运行结果
- `/status --short`: 查看适合终端和脚本消费的短状态
- `/events [lines]`: 查看最近的结构化事件

## 调试工作区

AI 终端和人工排查都可以用调试工作区隔离原始 plan：

```powershell
/use .\test-plans\regression\failure-capture\plan.json
/debug prepare latest-failure
/use .\test-plans\basic\file-io\plan.json
/debug create file-io-investigation
/debug list
/debug inject print,variables file-io-investigation
/debug patch file-io-investigation
/artifacts debug
```

工作区创建在当前 plan 包：

```text
output/debug/<timestamp>-<name>/
  source-copy/
  injected-plan/
  notes.md
  report.md
  patch.diff
  manifest.json
```

`source-copy/` 是原始 plan 快照；`injected-plan/` 用于注入调试用的 `print`、截图、HTML、变量快照和 `manual_confirm`。

`debug prepare [name]` 会读取最近失败运行的截图、HTML、页面状态和 DOM 摘要，创建 debug workspace，并把诊断步骤插到失败 step 之前。这样调试副本可以在失败动作执行前再次保存页面状态，避免把诊断步骤追加到永远执行不到的末尾。

支持的注入 preset：

- `print`: 在运行日志中写入调试检查点。
- `variables`: 写出变量快照到 `output/variables/debug/`。
- `manual_confirm`: 在当前可见 Playwright 浏览器窗口中等待用户完成登录、验证码、二次验证或页面检查。
- `screenshot`: 截图，需要指定浏览器会话名。
- `html`: 保存页面 HTML，需要指定浏览器会话名。

一次性注入命令：

```powershell
python .\main.py plan debug-inject --workspace .\test-plans\basic\file-io\output\debug\<run> --preset print --preset variables --message "debug checkpoint"
python .\main.py plan debug-inject --workspace .\test-plans\basic\file-io\output\debug\<run> --preset screenshot --preset html --browser demo --position before_step --step 3
python .\main.py plan debug-fix --workspace .\test-plans\regression\failure-capture\output\debug\<run> --hint "系统账户"
python .\main.py plan debug-fix --workspace .\test-plans\regression\failure-capture\output\debug\<run> --hint "系统账户" --apply
python .\main.py plan debug-patch --workspace .\test-plans\basic\file-io\output\debug\<run>
```

运行调试副本：

```powershell
python .\main.py plan run --file .\test-plans\basic\file-io\output\debug\<run>\injected-plan\plan.json --run-name injected-debug
```

使用结构化工具完成调试副本修复闭环：

```powershell
python .\main.py tool call read_debug_workspace --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>"}'
python .\main.py tool call propose_debug_fix --args-json '{"workspace":".\test-plans\regression\failure-capture\output\debug\<run>","user_hint":"系统账户"}'
python .\main.py tool call propose_debug_fix --args-json '{"workspace":".\test-plans\regression\failure-capture\output\debug\<run>","user_hint":"系统账户","apply":true,"run_after_apply":true}'
python .\main.py tool call patch_debug_workspace_json --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>","root":"injected-plan","relative_path":"plan.json","operations":[{"op":"replace","path":["steps",0,"message"],"value":"debug message"}]}'
python .\main.py tool call write_debug_workspace_file --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>","root":"injected-plan","relative_path":"plan.json","content":"..."}'
python .\main.py tool call validate_debug_plan --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>"}'
python .\main.py tool call run_debug_plan --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>","run_name":"injected-debug"}'
python .\main.py tool call generate_debug_patch --args-json '{"workspace":".\test-plans\basic\file-io\output\debug\<run>"}'
```

确认修复后应用补丁：

```powershell
python .\main.py plan debug-apply --workspace .\test-plans\basic\file-io\output\debug\<run> --yes
```

不带 `--yes` 不会修改原始 plan 包。

`propose_debug_fix` 当前只做保守的候选修复，第一版覆盖 selector 超时：从失败 HTML 的 DOM 摘要中提取候选 selector，并结合用户 hint 排序。它应用候选时会先把 `source-copy/plan.json` 复制回 `injected-plan/plan.json`，再写入最小修复，避免调试注入污染最终 patch。
