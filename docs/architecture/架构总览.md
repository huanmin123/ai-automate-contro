# 架构总览

## 项目定位

这是一个可长期演进的 Playwright + Python 自动化框架。

目标不是“只写一个脚本完成一个网站操作”，而是沉淀一套以 plan 包为边界、可维护的自动化执行内核。它需要支持：

- 多浏览器协作
- 外部数据源驱动
- 条件分支与循环
- 日志与失败排查
- 文档化维护

## 当前架构分层

### 1. 入口层

- `main.py`

职责：

- 读取 JSON 文件
- 校验并执行 `plan.json`

### 2. 执行核心层

- `src/keygen_automation/executor.py`
- `src/keygen_automation/actions.py`
- `src/keygen_automation/runtime.py`
- `src/keygen_automation/conditions.py`
- `src/keygen_automation/template.py`

职责：

- 管理执行期变量
- 管理浏览器会话
- 管理同一会话下的多个页面
- 执行动作组件
- 解析变量模板
- 处理条件判断与循环

### 3. 文档与示例层

- `handbook/`
- `plans/minimal-browser-plan/`
- `test-plans/basic/`
- `test-plans/data-driven/`
- `docs/`

职责：

- 面向使用者提供组件手册
- 提供稳定的本地 HTML 测试夹具
- 提供可运行示例资产
- 沉淀项目设计与演进记录

## Plan 包执行模型

一个 plan 包就是一个完整需求，主入口固定为 `plan.json`。如果流程较长，只能拆到当前包的 `sub-plans/*-plan.json`，不能引用其他需求包的 `plan.json`。

推荐结构：

```text
test-plans/<category>/<requirement>/
  plan.json
  config.json
  sub-plans/
  resources/
  output/
  docs/
```

`plans/` 只放公开的最小参考 plan 包；项目内部验证和回归用例放在 `test-plans/`。`test-plans/` 下面直接按类别放 plan 包，不再引入 `suites/`、`workspaces/` 或额外的 `plans/` 层级。

## 输出结构

当前运行产物落在对应 plan 包的 `output/`。截图、录屏、下载、HTML、JSON、CSV、TXT、storage state、失败截图、失败 HTML 和失败页面状态都必须留在这个目录下，并按组件写入固定分区。

每次执行都会在 `output/` 下创建独立运行目录，里面至少包含：

- `state.json`
- `result.json`
- `report.md`
- `run.log`
- `events.jsonl`
- 交互式运行的 `commands.jsonl`

失败场景还会额外包含：

- `failure-screenshots/`
- `failure-html/`
- `failure-page-state/`

AI 终端的失败分析工具会基于 `failure-html/` 生成 DOM 摘要，提取交互元素和 selector 提示；证据仍不足时，才在 `output/debug/` 创建调试副本并运行真实浏览器诊断。

## 示例资产规则

示例与回归计划应优先依赖本地夹具页面，而不是外部真实网站或公网接口。

这样可以保证：

- 测试可重复
- 回归稳定
- 不受站点改版影响
- 更适合长期维护
