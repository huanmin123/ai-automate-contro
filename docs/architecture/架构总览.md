# 架构总览

## 项目定位

这是一个可长期演进的 Playwright + Python 自动化框架。

目标不是“只写一个脚本完成一个网站操作”，而是沉淀一套可以跨项目、跨场景、跨测试任务复用的自动化执行平台。它需要支持：

- 单计划执行
- 多浏览器协作
- 多项目套件执行
- 顺序与并发调度
- 外部数据源驱动
- 条件分支与循环
- 日志与失败排查
- 文档化维护

## 当前架构分层

### 1. 入口层

- `main.py`

职责：

- 读取 JSON 文件
- 识别当前是 `plan` 还是 `suite`
- 将执行请求分发到不同执行器

### 2. 执行核心层

- `src/keygen_automation/executor.py`
- `src/keygen_automation/actions.py`
- `src/keygen_automation/runtime.py`
- `src/keygen_automation/conditions.py`
- `src/keygen_automation/template.py`

职责：

- 管理执行期变量
- 管理浏览器会话
- 管理同一会话下的多个页面
- 执行动作组件
- 解析变量模板
- 处理条件判断与循环

### 3. 套件调度层

- `src/keygen_automation/suite.py`

职责：

- 让多个项目或多个场景可以统一编排
- 支持顺序执行和并发执行
- 为每个子计划注入独立变量
- 生成每轮 suite 的统一输出目录

### 4. 文档与示例层

- `handbook/`
- `examples/scenarios/basic/`
- `examples/scenarios/data-driven/`
- `examples/suites/smoke/`
- `examples/suites/regression/`
- `examples/data/`
- `docs/`

职责：

- 面向使用者提供组件手册
- 提供可运行示例资产
- 沉淀项目设计与演进记录

## 多项目测试执行模型

### 顺序执行

适合：

- 多个项目共用一套有限资源
- 前一个项目的结果会影响后一个项目
- 需要稳定、可控、便于观察

优点：

- 日志更清晰
- 资源冲突更少
- 排查问题更直接

缺点：

- 总耗时更长

### 并发执行

适合：

- 三个项目彼此独立
- 机器性能足够
- 你更关心整体吞吐量

优点：

- 总耗时更短
- 更适合批量回归

缺点：

- CPU、内存、网络占用更高
- 日志和故障排查更复杂
- 如果多个项目依赖同一个外部资源，容易互相干扰

### 当前建议

默认先用顺序执行，把计划跑稳定。

等每个项目的单计划足够稳、日志也足够清晰后，再把适合独立运行的项目切到并发模式。也就是说：

- 开发调试阶段：优先顺序
- 批量回归阶段：再考虑并发

## 输出结构

当前运行产物主要落在：

- `output/plan-runs/`
- `output/suite-runs/`

每次执行都有自己的目录，里面至少包含：

- `run.log`
- `events.jsonl`
- `result.json`

suite 级输出还包含：

- `suite-summary.json`
- `suite-summary.md`

失败场景还会额外包含：

- `failure-screenshots/`
