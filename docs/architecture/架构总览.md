# 架构总览

## 项目定位

这是一个可长期演进的 Playwright + Python 自动化框架。

目标不是“只写一个脚本完成一个网站操作”，而是沉淀一套以 plan 包为边界、可维护的自动化执行内核。它需要支持：

- 多浏览器协作
- 外部数据源驱动
- 条件分支与循环
- 日志与失败排查
- 文档化维护

## 当前架构分层

### 1. 入口层

- `main.py`
- `src/ai_automate_contro/app/cli.py`
- `src/ai_automate_contro/app/parser.py`
- `src/ai_automate_contro/client/`
- `src/ai_automate_contro/app/command_helpers.py`

职责：

- 提供极薄启动入口
- 解析 CLI 子命令
- `main.py`/`aic` 分发 AI、tool、self-check 命令；`cplan.py`/`cplan` 分发 list/create/validate/run/schedule/debug-* 和 plan self-check
- 提供 Textual AI-first 交互客户端

### 2. 执行核心层

- `src/ai_automate_contro/engine/executor.py`
- `src/ai_automate_contro/engine/actions/`
- `src/ai_automate_contro/engine/runtime.py`
- `src/ai_automate_contro/engine/conditions.py`
- `src/ai_automate_contro/engine/template.py`

职责：

- 管理执行期变量
- 管理浏览器会话
- 管理同一会话下的多个页面
- 执行动作组件
- 解析变量模板
- 处理条件判断与循环

动作模块按行为域维护，例如 browser flow、browser input/capture/events/dialog、control flow、files、assertions、extraction 和 AI task。拆分以职责边界和回归风险为准，不以文件大小本身为目标。

### 3. Plan 与 AI 支撑层

- `src/ai_automate_contro/plans/`
- `src/ai_automate_contro/ai/`
- `src/ai_automate_contro/debug/`
- `src/ai_automate_contro/support/`

职责：

- plan 加载、校验、包发现和运行产物读取
- 受控专项 AI action 和 plan 级 AI 终端
- debug workspace、patch 生成和人工批准后的补丁应用
- 日志与通用工具函数

AI 终端工具由 `terminal_tool_registry.py` 的 `AI_TERMINAL_TOOL_SPECS` 单表维护处理函数、参数 schema、描述、`project_root` 注入和保护标记；工具实现按 plan、运行产物分析、debug workspace、AI service 调用等职责放在相邻模块中。受保护的原始 plan 补丁应用必须经过 HITL 中断和用户 `/approve`。

### 4. 文档与示例层

- `handbook/`
- `plans/minimal-browser-plan/`
- `test-plans/` 源码开发回归夹具
- `docs/`

职责：

- 面向使用者提供组件手册
- 提供稳定的本地 HTML 测试夹具
- 提供可运行示例资产
- 沉淀项目设计与演进记录

## Plan 包执行模型

一个 plan 包就是一个完整需求，主入口固定为 `plan.json`。如果流程较长，只能拆到当前包的 `sub-plans/*-plan.json`，不能引用其他需求包的 `plan.json`。

推荐结构：

```text
plan-package/
  plan.json
  config.json
  sub-plans/
  resources/
  output/
  docs/
```

运行根通过 `plan.config.plan_roots` 决定可发现的 plan 集合。发行包默认使用 `plans/`；源码开发仓库中的 `test-plans/` 只作为开发回归夹具。

## 输出结构

当前运行证据和中间产物落在对应 plan 包的 `output/`。截图、录屏、下载、HTML、JSON、CSV、TXT、storage state、失败截图、失败 HTML 和失败页面状态都必须留在这个目录下，并按组件写入固定分区。AI 终端的最终交付层可以在运行成功后使用 `export_local_file` 把整理后的结果写入用户明确指定的本机路径，例如 Downloads、桌面或绝对文件路径。

每次执行都会在 `output/` 下创建独立运行目录，里面至少包含：

- `state.json`
- `result.json`
- `report.md`
- `run.log`
- `events.jsonl`
- 交互式运行的 `commands.jsonl`

失败场景还会额外包含：

- `failure-screenshots/`
- `failure-html/`
- `failure-page-state/`

AI 终端的失败分析工具会基于 `failure-html/` 生成 DOM 摘要，提取交互元素和 selector 提示；证据仍不足时，才在 `output/debug/` 创建调试副本并运行真实浏览器诊断。

## 示例资产规则

示例与回归计划应优先依赖本地夹具页面，而不是外部真实网站或公网接口。

这样可以保证：

- 测试可重复
- 回归稳定
- 不受站点改版影响
- 更适合长期维护
