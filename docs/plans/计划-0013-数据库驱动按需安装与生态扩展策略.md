# 计划-0013-数据库驱动按需安装与生态扩展策略

## 背景

数据库 common action 已经支持 SQLite、DuckDB、MySQL/MariaDB、PostgreSQL、Oracle 和 Redis。后续还可能继续扩展 SQL Server、MongoDB、ClickHouse、向量库或消息队列类数据源。

这些能力应该在 action 和 plan 契约上可用，但不能让默认安装被所有数据库驱动拖大。用户只做浏览器、桌面或本地 SQLite 自动化时，不应安装 MySQL、PostgreSQL、Oracle、Redis 等驱动。

## 目标

- 默认安装不包含任何服务型数据库驱动。
- SQLite 继续作为零额外依赖的本地化默认方案。
- 服务型数据库驱动按库拆分为独立 optional extra。
- 运行时继续懒加载驱动，缺依赖时给出对应库的安装命令。
- handbook、功能设计和验证说明都明确“支持不等于默认安装”。
- 后续新增数据库生态时，先落 action 能力，再新增对应 extra 和文档。

## 范围

- 拆分 `pyproject.toml` 数据库 optional dependency。
- 更新 SQL/Redis 运行时缺驱动提示。
- 更新数据库 handbook、功能设计、测试说明和数据库示例说明。
- 保留显式聚合 extra，供用户明确需要多库能力时使用。

## 不做

- 不把所有数据库驱动放进默认依赖。
- 不自动安装用户机器缺失的数据库驱动。
- 不为尚未落地的数据库能力预先增加可安装 extra。
- 不新增 `automation_type: "database"`。
- 不把 SQL、Redis、MongoDB、Elasticsearch 强行合并成一个 action。

## 设计方向

### 1. Optional extras 分层

当前可用的独立安装入口：

```powershell
pip install -e '.[db-postgresql]'
pip install -e '.[db-mysql]'
pip install -e '.[db-redis]'
pip install -e '.[db-oracle]'
pip install -e '.[db-duckdb]'
```

聚合入口只给明确需要多库能力的用户：

```powershell
pip install -e '.[database-local]'
pip install -e '.[database-sql]'
pip install -e '.[database-basic]'
pip install -e '.[database-all]'
```

含义：

- `db-postgresql`: 只安装 PostgreSQL 驱动。
- `db-mysql`: 只安装 MySQL/MariaDB 驱动。
- `db-redis`: 只安装 Redis 驱动。
- `db-oracle`: 只安装 Oracle 驱动。
- `db-duckdb`: 只安装 DuckDB 驱动。
- `database-local`: 当前本地分析数据库驱动聚合。
- `database-sql`: 当前 SQL 数据库驱动聚合。
- `database-basic`: MySQL、PostgreSQL、Redis 常用组合。
- `database-all`: 当前已支持数据库驱动全量组合。

### 2. 懒加载原则

action 模块可以存在于项目中，但第三方驱动只能在真正执行对应连接时导入：

- `sql.connection.type=sqlite`: 使用标准库 `sqlite3`。
- `sql.connection.type=postgresql`: 执行时导入 `psycopg`。
- `sql.connection.type=mysql`: 执行时导入 `pymysql`。
- `sql.connection.type=oracle`: 执行时导入 `oracledb`。
- `sql.connection.type=duckdb`: 执行时导入 `duckdb`。
- `redis` action: 执行时导入 `redis`。

缺少驱动时，错误信息必须指向对应 extra，不能泛化成安装所有数据库依赖。

### 3. 后续生态扩展规则

新增数据库能力时按这个顺序落地：

1. 明确 action 边界和返回 payload。
2. 增加最小运行时实现，并保持懒加载。
3. 增加独立 extra，例如 `db-sqlserver`、`db-mongodb`。
4. 增加 handbook 和功能设计说明。
5. 增加默认可离线回归或显式真实服务回归。
6. 只有在驱动已经可用且 action 已落地后，才加入聚合 extra。

建议优先级：

- SQL Server：企业环境常见，优先评估 ODBC Driver 安装前置条件。
- DuckDB：已按本策略接入，后续继续补 CSV/Parquet/Excel 专项示例。
- MongoDB：数据模型不同，应新增独立 `mongo` common action。
- 大结果 cursor/stream：服务于百万级查询导出和分批传递。
- SQL copy/sync：从一个数据源查询后批量写入另一个库。

## 实施步骤

1. 拆分数据库 optional extras，移除单一 `database` 推荐入口。
2. 修改 SQL/Redis 缺依赖提示，按库输出安装命令。
3. 更新 SQL/Redis handbook。
4. 更新数据库连接功能设计。
5. 更新数据库测试计划说明和验证说明。
6. 跑 `compileall`、SQLite 数据库组件自检和 release matrix 相关项。

## 风险

- 用户可能继续使用旧的 `pip install -e .[database]` 习惯，需要通过文档和错误提示引导到独立 extra。
- 聚合 extra 仍可能被用户显式选择；这是明确选择，不是默认安装。
- SQL Server、Oracle、DuckDB 等驱动包大小和系统前置条件差异大，必须在各自能力落地时单独说明。

## 验收标准

- 默认 `pip install -e .` 不安装服务型数据库驱动。
- 缺 PostgreSQL/MySQL/Oracle/Redis 驱动时，报错提示对应 `db-*` extra。
- SQLite 自检不依赖任何额外数据库包。
- 文档不再推荐 `pip install -e .[database]` 作为通用数据库安装命令。
- 后续扩展计划明确使用“独立 extra + 懒加载 + 显式回归”的模式。
