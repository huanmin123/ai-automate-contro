# 计划-0003-AI终端与交互式执行器重构

## 背景

当前 plan 包模型已经明确：一个 `plan.json` 是一个独立需求，运行产物必须留在当前 plan 包 `output/`。下一步需要补齐 CLI 交互能力，并把 AI 放到 plan 生命周期管理的位置。

新的方向是：

- `main.py`/`aic` 是 AI-first 入口。
- `cplan.py`/`cplan` 是无 AI 的确定性 plan 管理入口。
- Textual 客户端负责 AI 对话、工作计划、队列、工具进度、审批和错误显示。
- AI 后端负责 plan 级协作，并通过结构化工具调用确定性能力。
- 专项 AI 作为受控数据处理能力存在。

## 目标

- 建立 Textual AI-first 交互客户端。
- 建立可被终端控制的交互式执行器。
- 建立 AI 终端第一版工作流。
- 建立专项 AI action 的统一规范。
- 保持所有运行产物只写入当前 plan 包 `output/`。

## 范围

### 第一阶段：文档和边界

- 更新 AI 架构设计。
- 更新 AI 功能设计。
- 明确 Textual 客户端、plan 管理能力和 AI 后端职责。
- 明确专项 AI 的配置、schema、输出和安全边界。

### 第二阶段：统一终端与管理模式

- 基于 Textual 的 AI-first 交互客户端：当前默认入口直接进入对话式客户端；确定性 plan/tool/self-check 能力走一次性 CLI 子命令。
- `list`：已完成第一版，按当前运行根的 `plan_roots` 发现 plan 包。
- `use`：已完成第一版，选择当前 plan。
- `run`：已完成第一版，先校验再运行当前 plan。
- `status`：当前由 Textual 客户端统一展示客户端、AI 会话和当前 plan 状态；输出目录、日志和事件排查优先交给 AI 工具或一次性 CLI 命令，不再作为常用交互命令展示。
- plan 模式命令提示：已完成当前版，只有行首 `/command` 会作为命令；普通文字和文字中间的 `/xxx` 不触发命令。
- 无 AI 的一次性 `cplan create` / `cplan validate` / `cplan run`：已完成第一版，供脚本和回归调用。

### 第三阶段：交互式执行器

- `manual_confirm` 等待态：已完成第一版，Textual 客户端里的 AI 通过自然语言继续/停止；无 AI 环境由 `cplan run` 在当前命令行等待。
- `state.json`：已完成第一版，记录 running / waiting / passed / failed、当前 step、等待信息、错误和输出目录。
- 运行状态：已完成第一版，Textual 客户端优先读取 `state.json`。
- `events.jsonl`：已完成第一版，结构化事件写入输出目录；Textual 客户端显示关键进度，完整事件通过 AI 工具或一次性 CLI 排查。
- 命令队列：已完成第一版审计结构，AI 对话里的继续/停止通过事件和自然语言处理，不再把 plan 管理命令混进 Textual 常用候选。
- 输出报告结构：已完成第一版，每次运行生成 `report.md`。

### 第四阶段：AI 终端

- 基于 LangChain Agent + LangGraph 的 plan 级工作流：已切换为 `langchain.agents.create_agent` + 原生 `tool_calls`。
- 工具封装：已完成 LangChain `StructuredTool` 包装，并已切换为共享显式 Pydantic 参数模型；`python .\main.py tool call` JSON 入口复用同一套 schema 校验，用于脚本和回归。
- 工具注册表一致性：已完成 `python .\main.py tool check`，检查工具函数、参数 schema 和工具描述是否完全对齐，并在工具调用和 LangChain tool 构建前执行。
- LangChain 工具构建自检：已完成 `python .\main.py self-check ai-tools`，覆盖 `StructuredTool` 构建、共享 Pydantic schema、工具 invoke 回调和受保护工具 HITL 守卫。
- 会话持久化：已完成 LangGraph `SqliteSaver` 接入，AI 终端通过 `--thread` 恢复同一会话，checkpoint 存放在本地 `.keygen/`。
- 线程业务上下文：已完成 `current_plan_path`、`current_debug_workspace`、`latest_output_dir` 状态字段和 middleware 注入，终端命令与工具返回会维护这些状态。
- 用户确认节点：已接入 LangChain `HumanInTheLoopMiddleware`，`apply_debug_patch_after_approval` 会在工具执行前中断，用户通过 `/approve` / `/reject` 恢复图。
- 失败分析和补丁建议：已完成第一版 `analyze_latest_run_failure` 证据汇总工具，可读取状态、报告、日志、事件、命令审计、失败截图、失败 HTML、失败页面状态和失败 DOM 摘要。
- 证据不足后的真实调试策略：已记录，AI 只能在 debug workspace 的 `injected-plan/` 中注入诊断并运行调试副本，不能直接污染原始 plan。
- 真实只读工具回归：已通过 `real-tool-regression-validate-plan` 线程验证模型会用原生 `tool_calls=validate_plan` 调用工具，结果进入 `ToolMessage`，同一线程由 SQLite checkpoint 保存历史。

### 第六阶段：AI 调试修复

- `output/debug/<run>/source-copy/` 原始 plan 快照：已完成第一版工具。
- `output/debug/<run>/injected-plan/` 调试注入副本：已完成第一版工具。
- `debug create` / `debug list`：Textual 客户端不再承载无 AI plan 管理；无 AI 调试统一使用 `cplan debug-*`，AI 端通过结构化工具完成。
- `debug prepare` / `prepare_failure_debug_workspace`：已完成基于最近失败自动创建调试工作区并在失败步骤前注入诊断。
- `debug fix` / `propose_debug_fix`：已完成 selector 超时的第一版候选修复，可基于 DOM 摘要生成干净 patch。
- selector 修复门禁：已完成无提示歧义保护；没有明确目标提示时只返回候选，不自动写入 `injected-plan/`。
- `cplan debug-create`：已完成一次性命令，供无 AI 调试调用。
- `artifacts [filter] [limit]`：已完成第一版输出产物浏览命令。

- 经用户确认后应用到原始 plan：已完成第一版 `--yes` 显式确认命令。

## 不做

- 不恢复批量套件或工作区中间层。
- 不让主 `plan.json` 互相引用。
- 不让 AI 自动绕过验证码或真人验证。
- 不让 LangGraph 接管浏览器执行器。

## 验收标准

- 用户可以只通过 CLI 创建、校验、运行 plan。
- plan 运行中可以等待用户确认并继续。
- AI 终端能读取 plan、运行 plan、分析失败、提出补丁。
- AI 终端应用补丁前必须请求用户确认。
- 所有运行产物仍在当前 plan 包 `output/` 下。
- 专项 AI 产物统一写入 `output/ai/`。
- AI 调试修复不污染原始 plan，调试副本和报告统一写入 `output/debug/`。

## 风险

- 交互式执行器会改变当前同步执行模型，需要小步迁移。
- AI 终端如果直接写文件，容易产生不可控变更，因此必须通过补丁和确认。
- 专项 AI schema 如果设计过宽，会削弱可重复性和可调试性。
