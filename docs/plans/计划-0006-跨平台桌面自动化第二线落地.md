# 计划-0006-跨平台桌面自动化第二线落地

## 背景

浏览器自动化主线已经覆盖 Playwright 会话、页面交互、网络事件、截图、断言、文件读写、HTTP 客户端、受控本地命令和失败现场。下一阶段需要把能力扩展到真实 Windows/macOS 桌面会话，让 plan 尽量覆盖人像平时操作电脑一样完成的本机操作，包括打开 App、窗口切换、菜单点击、键盘、鼠标、拖拽、截图、控件定位、图像识别和 OCR。

桌面自动化的稳定性低于 Playwright。它不能只靠坐标点击实现，也不能直接塞进 `command` action。应建立独立的 desktop session/backend/action 体系，并让 Windows 和 macOS 差异停留在 backend adapter 内。

## 目标

- 新增桌面自动化第二执行线，统一 JSON plan 包模型，但通过顶层 `automation_type` 与浏览器线完全隔离。
- 主 `plan.json` 必须声明 `automation_type: "browser"` 或 `automation_type: "desktop"`，执行器按类型选择解析器、runtime manager 和 action registry。
- 桌面线目标是控制真实桌面会话中的 App 和系统窗口，不限于终端命令，也不限于某一个固定 App。
- 第一阶段覆盖 App 启动、窗口列表、窗口聚焦、窗口关闭/最小化/最大化/还原、控件枚举/定位/读取/点击、快捷键、文本输入、坐标点击/双击/右键/滚动/拖拽、截图、桌面状态快照和等待窗口。
- Phase 0 先使用轻量 `native` backend：Windows 通过 `ctypes` 做窗口枚举、聚焦和生命周期控制，macOS 通过 AppleScript/System Events 做基础窗口查询、聚焦和生命周期控制，截图使用 Pillow `ImageGrab`，键鼠输入使用可选 `pyautogui`。
- `pywinauto`、`mss`、多显示器校准和更完整视觉能力保留为后续增强；`desktop_vision type=locate_image` 已作为 OpenCV 模板匹配能力落地。复杂控件、窗口/控件截图、`desktop_vision source_target=window/element` 和 `desktop_vision type=locate_text` 基础 OCR 已在计划-0007 中继续落地。
- 输出证据继续写入当前 plan 包 `output/`。
- 真实桌面应用 plan 创建前必须先获取窗口、控件、截图或人工确认等证据，不能凭文字猜坐标。

## 范围

- 新增 `DesktopSession` 和 desktop backend adapter。
- 按 [桌面Action与Runtime契约](../functions/桌面Action与Runtime契约.md) 固定 Phase 0 action schema、payload、输出分区、失败证据和 self-check。
- 新增 `automation_type` 顶层字段校验、子计划继承规则和跨线 action 拒绝规则。
- 新增桌面 action 执行模块和 action 注册。
- 新增桌面输出分区和失败现场采集。
- 新增校验规则、handbook 文档和 test-plans。
- 新增 `desktop-components` 综合自检和 `desktop-real-app` 显式真实 App 回归入口。
- 同步更新架构、功能、开发验证和 AI 协作规则。

## 不做

- 不把桌面控制并入 `command` action。
- 不允许同一个普通 plan 同时混用浏览器专属 action 和桌面专属 action。
- 不通过 action 名称猜测 plan 类型；缺少 `automation_type` 的主 plan 必须校验失败。
- 不把 `PyAutoGUI` 坐标级鼠标输入作为最终主方案；Phase 0 已支持坐标点击/双击/右键/滚动/拖拽用于基础闭环，后续稳定控件操作仍应优先 UIA/AX 或视觉 locator。
- 不把图像识别/OCR 作为第一定位方式。
- 不在第一阶段实现完整 OCR、图像模板管理、多显示器坐标校准和录屏 trace。
- 不默认接入 WinAppDriver/Appium、FlaUI sidecar 或 RPA Framework。
- 不绕过 Windows UAC、安全桌面、管理员权限或 macOS TCC 隐私授权。
- 不自动替用户点击 macOS 隐私授权。
- 不承诺锁屏、无 GUI、远程会话断开、受保护内容、银行安全控件、反作弊游戏、人机验证或安全策略可被自动化绕过。

## 关键设计

### Action 分组

桌面 action 应覆盖人类常见电脑操作，而不是只覆盖终端命令。动作模型按能力域拆分：

- 窗口和 App：启动、连接、列窗口、聚焦、移动、调整大小、最小化、最大化、关闭。
- 输入：文本、快捷键、鼠标点击、双击、右键、拖拽、滚轮、剪贴板。
- 控件：点击、设置文本、读取文本、选择项、等待状态、断言状态。
- 视觉：截图、区域截图、图像定位、OCR 定位、标注失败证据。
- 人工交接：权限授权、验证码、二次验证、系统安全边界和无法可靠判断的 UI。

顶层类型：

```json
{
  "automation_type": "desktop",
  "variables": {},
  "steps": []
}
```

`automation_type=browser` 只能使用浏览器专属 action 和通用 action；`automation_type=desktop` 只能使用桌面专属 action 和通用 action。子计划继承主 plan 类型，不能跨线。

### macOS 权限流程

macOS backend 初始化时应检测：

- Accessibility
- Screen Recording
- Automation
- Input Monitoring，只有录制或监听全局输入时需要

缺权限时执行器应：

1. 记录缺失权限和启动主体。
2. 触发系统授权提示或打开系统设置页。
3. 进入人工确认等待。
4. 用户授权后继续执行。

开发阶段不强制 Apple Developer ID。正式分发时应打包成固定 `.app`，使用稳定 bundle id、Developer ID 签名和 notarization，减少 Gatekeeper 和重复授权问题。

## 验收标准

- 文档评审后，action 命名、字段风格和目录边界与现有 plan/action 体系一致。
- `automation_type`、handbook browser/desktop 双入口、执行器 action registry 分发和 validator 跨线拒绝规则都落地。
- `command` action 边界不变，不承担桌面窗口、App 启动或坐标级键鼠输入。
- Windows MVP 能在真实桌面会话下通过 `desktop_app` 启动并跑通 Notepad 最小 plan。
- macOS MVP 能检测并引导 Accessibility / Screen Recording / Automation 授权，授权后通过 `desktop_app` 启动并跑通 TextEdit 最小 plan。
- 所有桌面运行产物只写当前 plan 包 `output/`。
- 无 GUI、锁屏、权限不足或依赖缺失时，自检返回 `skipped` 和原因，不误报为框架失败。
- 失败时至少保存桌面截图、窗口列表、backend 信息、权限状态和目标定位摘要。
- handbook、功能文档、架构文档和验证说明同步更新。

## 风险

- “人能操作”不等于“自动化一定稳定”。桌面控制需要目标 UI 可见、会话未锁屏、权限已授权、控件树或视觉特征可识别。
- macOS 权限绑定到启动主体，从 Terminal、VS Code 和打包 `.app` 启动会产生不同授权对象。
- macOS AX 控件树不如 Windows UIA 稳定，缺少稳定 identifier 时需要 fallback。
- Windows UAC、安全桌面和管理员权限窗口不能被普通权限进程稳定控制。
- 坐标级鼠标输入和图像识别受 DPI、多显示器、Retina、主题和字体渲染影响。
- OCR 依赖本机语言包和图像质量，不能作为第一阶段主定位方式。
- 桌面自动化真实回归依赖交互桌面，CI 中需要跳过或使用专门 runner。
