# 计划-0010-数据库真实服务回归与扩展优化

## 背景

`sql` 和 `redis` common action 第一阶段已经覆盖 SQLite、本地 SQL 执行、批量写入、查询结果落盘，以及 MySQL、PostgreSQL、Oracle、Redis 的基础驱动接入。用户也明确提出 SQLite 应作为很多自动化用户的本地化优先选择，这个判断应继续保留。

下一阶段的重点不是再引入一个新的执行线，而是把数据库能力从“能用”推进到“可长期回归、可被 AI 稳定生成、可处理更多真实业务数据量和数据库差异”。

## 目标

- 新增数据库组件级自检入口，默认只跑 SQLite，保证离线、确定性、无需中间件。
- 增加真实服务可选回归，覆盖 MySQL、PostgreSQL、Redis、Oracle，以及通过 `http` action 覆盖 Elasticsearch。
- 补齐大批量数据、分页查询、事务、临时表清理和幂等写入的设计与回归。
- 让 AI 生成 plan 时优先选择 SQLite；只有用户明确给出连接信息或业务需要服务型数据库时，才生成 MySQL、PostgreSQL、Oracle、Redis 等连接。
- 保持 browser/desktop 两条执行线隔离，数据库能力继续作为 common action 被两条线共享。

## 范围

- 新增 `python .\cplan.py self-check database-components`。
- 新增 `test-plans/database/` 下的真实服务示例包，但示例默认只使用占位连接名或本机私有配置引用，不把真实密码写入公开示例。
- 在验证说明中加入数据库回归矩阵和发布前可选项。
- 扩展 `sql` action 的事务、批量、分页和 Oracle 差异处理。
- 明确 Elasticsearch 暂时继续使用 `http` action；只有 API 使用频率明显增加时，再考虑单独的 `elasticsearch` common action。

## 不做

- 不新增 `automation_type: "database"`。
- 不把 SQL、Redis、MongoDB、Elasticsearch 强行塞进一个通用 `database` action。
- 不内置 ORM、迁移系统、后台连接池或数据库服务启动器。
- 不在默认自检中依赖用户机器存在 MySQL、PostgreSQL、Oracle、Redis 或 Elasticsearch。
- 不自动脱敏本地调试输出；公开示例和文档使用占位值，真实本机配置由用户决定是否提交。

## 设计方向

### 1. 数据库组件自检

默认命令：

```powershell
python .\cplan.py self-check database-components
```

默认只执行 SQLite 回归：

- 创建本地 SQLite 文件。
- 建表、插入、批量写入、查询、更新、单值查询。
- 验证 `rows_path`、`result_path` 和 `output/sql/` 约束。
- 清理临时输出，不依赖网络或外部服务。

真实服务可选命令建议：

```powershell
python .\cplan.py self-check database-components --include-real-db --database-config .\local\database-services.json
```

可选真实服务矩阵：

| 类型 | 验证内容 | 写入策略 |
| --- | --- | --- |
| MySQL/MariaDB | 连接、版本查询、参数绑定、临时表批量写入 | 临时表或独立测试 schema |
| PostgreSQL | 连接、参数绑定、事务、临时表批量写入 | 临时表或事务回滚 |
| Redis | `PING`、`SET/GET`、Hash、List、Pipeline、TTL | 独立 key 前缀并清理 |
| Oracle | 连接、参数绑定、临时表批量写入、MERGE 示例 | 临时表并清理 |
| Elasticsearch | `http` action 查询集群信息和基础搜索 | 只读优先 |

### 2. 示例 plan 包

新增示例建议：

- `test-plans/database/mysql-basic/`
- `test-plans/database/postgresql-basic/`
- `test-plans/database/redis-basic/`
- `test-plans/database/oracle-basic/`
- `test-plans/http/elasticsearch-basic/`

这些示例不硬编码真实账号密码，优先从 `config.json.connections`、环境变量或 `--database-config` 指向的本机配置读取。

### 3. 事务与幂等写入

事务优先采用作用域式 action，而不是让用户手动拼 `begin/commit/rollback`：

```json
{
  "action": "sql",
  "type": "transaction",
  "connection": "crm_pg",
  "steps": [
    {
      "type": "execute",
      "sql": "update orders set status = :status where id = :id",
      "params": {
        "status": "synced",
        "id": "{{order_id}}"
      }
    }
  ],
  "output": {
    "as": "tx_result"
  }
}
```

运行期语义：

- 内部步骤全部成功才提交。
- 任一步骤失败则回滚。
- 事务结果写入 `output/sql/`。
- SQLite、PostgreSQL、MySQL、Oracle 分别走各自驱动事务能力。

### 4. 大结果与批量数据

需要补齐：

- `bulk_insert.batch_size`，避免一次性构造过大的 SQL 或参数数组。
- `query.page_size` 或 `query.limit/offset` 辅助字段，用于大表分页。
- `rows_path` 支持 JSONL，适合大量行流式写入。
- 查询结果变量默认继续受 `max_rows` 保护，超过限制时引导用户落盘。
- Redis `pipeline` 增加分批执行和结果落盘，避免变量池过大。

### 5. 跨数据库差异

现状与后续策略：

- SQLite 继续作为默认本地数据库。
- MySQL/PostgreSQL 保留 `bulk_insert mode=upsert` 的通用能力。
- Oracle 不伪造通用 upsert，后续补 `MERGE` 示例或 `sql type=merge` 专项能力。
- SQL Server 后续可接入 `sql` action，但应先评估 `pyodbc` 驱动安装成本和 Windows/Linux 差异。
- MongoDB 数据模型不同，若要支持应新增 `mongo` common action，不混入 `sql`。
- Elasticsearch 当前由 `http` action 覆盖；只有需要 bulk index、scroll/search_after、mapping 管理时再新增专项 action。
- ClickHouse 可作为后续扩展候选。

## 实施步骤

1. 新增 `database-components` self-check，默认 SQLite，支持 `--include-real-db`、`--database-config` 和 `--only`。
2. 抽象真实服务测试配置读取，支持连接名、内联连接对象和环境变量引用。
3. 增加 MySQL、PostgreSQL、Redis、Oracle、Elasticsearch 的只读 smoke 测试。
4. 增加临时表或独立 key 前缀写入测试，确保每次测试可清理。
5. 为 `sql` 增加事务 action 契约和最小实现。
6. 为 `sql.bulk_insert` 增加 `batch_size`，并补齐 SQLite/MySQL/PostgreSQL/Oracle 回归。
7. 为大查询增加 JSONL 输出和分页建议，避免把大结果塞进变量池。
8. 补充 handbook、功能文档和测试说明中的数据库矩阵。
9. 将默认 SQLite 自检纳入 `release-matrix`，真实服务回归只在显式参数下执行。

## 风险

- 真实数据库服务可能不可达、账号过期、权限不足或网络抖动，因此不能放入默认确定性矩阵。
- 生产库写入风险高，真实服务回归必须默认只读；需要写入时使用临时表、测试 schema、事务回滚或独立 key 前缀。
- SQL Server、Oracle、ClickHouse 等驱动安装成本和平台差异较大，应在可选依赖中分组或给出清晰错误提示。
- 大数据量查询如果直接保存到变量，会拖慢执行器和报告生成，必须引导落盘。
- 通用 upsert 不能覆盖所有数据库语法，跨库能力要以明确支持矩阵为准。

## 验收标准

- `python .\cplan.py self-check database-components` 在无外部数据库时通过。
- `python .\cplan.py self-check release-matrix` 默认包含数据库 SQLite 自检且通过。
- 显式提供真实服务配置时，MySQL、PostgreSQL、Redis、Oracle 和 Elasticsearch 可分别运行并返回结构化结果。
- 所有真实服务写入测试都使用临时资源，并在成功或失败后尽力清理。
- `sql` 事务失败时能回滚，成功时能提交，并生成可追踪输出。
- `bulk_insert.batch_size` 能处理超过单批限制的数据。
- handbook 和功能文档清楚说明 SQLite local-first、真实服务可选回归、Elasticsearch 走 `http` action。
