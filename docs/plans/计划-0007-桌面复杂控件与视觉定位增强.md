# 计划-0007-桌面复杂控件与视觉定位增强

## 背景

`计划-0006` 已完成桌面第二执行线 Phase 0：桌面 session、窗口、基础控件、键鼠、截图、图像模板定位、失败证据、AI 终端桌面探测和真实模型回归已经跑通。下一阶段需要从“能操作基础窗口和控件”推进到“能稳定处理更真实的桌面业务界面”。

真实桌面业务软件常见难点是表格、树、菜单、滚动容器、自绘控件、DPI/多显示器和无法通过控件树读取的文字。后续能力必须继续保持浏览器/桌面执行线隔离，不能把复杂控件或视觉能力塞进 `command`，也不能让 AI 在缺少能力矩阵或运行证据时猜坐标。

长期功能契约见 [桌面复杂控件与视觉增强设计](../functions/桌面复杂控件与视觉增强设计.md)。

## 目标

- 扩展 `desktop_element`，支持表格、树、菜单、滚动容器等复杂控件。
- 扩展 `desktop_capture` 和 `desktop_vision`，支持窗口/控件相对截图、相对视觉定位和全局坐标回填。
- 新增 `desktop_vision type=locate_text` OCR locator，作为控件树和图像模板不可用时的取证兜底。
- 建立坐标校准模型，统一截图、控件 bounds、视觉命中和鼠标输入的坐标空间。
- 扩展 `capability_matrix`、AI 质量门禁和 self-check，让 AI 只在当前环境具备能力时生成高级桌面 plan。
- 增加 Windows 临时复杂控件夹具和真实跨 App 样例；macOS 复杂控件真机回归后续在 Mac 上补齐。

## 范围

已完成的首批实现范围：

- `desktop_element type=get_table/select_cell/get_tree/expand_tree/collapse_tree/select_tree/invoke_menu/scroll_element`。
- `desktop_element type=invoke_menu open_context_menu=true`。
- `desktop_capture target=window/element`，输出 `target`、`source_bounds`、`coordinate_space`、窗口/控件定位证据。
- `desktop_vision source_target=window/element`，输出屏幕全局 `bounds/point` 和 source 局部 `local_bounds/local_point`。
- Windows 临时 WinForms 夹具：DataGridView、TreeView、MenuStrip、ContextMenuStrip、滚动 Panel，覆盖表格读取、单元格选择、树读取、树节点展开/折叠/选择、菜单栏路径触发、右键上下文菜单路径触发、滚动容器语义滚动、输出 JSON、状态文本和 `desktop-annotations` 标注。
- validator、runtime、backend、AI 质量门禁、AI 终端提示词、handbook 和 `desktop-components` 回归同步。

后续第一批收尾范围：

- `capability_matrix` 继续扩展 screenshot/vision/coordinates。
- `desktop-advanced-components` 自检入口，或先并入 `desktop-components` 后再拆分。

第二批实现范围：

- `desktop_vision type=locate_text`，优先 Tesseract provider。
- OCR 输出 raw text、blocks、matches、bounds、confidence、source 和 annotation。
- OCR schema、handbook、AI 质量门禁和 self-check。

第三批实现范围：

- 多显示器、负坐标、高 DPI、Retina 坐标归一。
- 更真实跨 App 样例。
- macOS AX 复杂控件映射和真机回归。

不在本计划第一批实现：

- 完整录屏 trace。
- Windows Graphics Capture 实时流。
- FlaUI/.NET sidecar。
- Appium/WinAppDriver。
- 自动绕过 UAC、安全桌面、锁屏、macOS TCC 授权或高权限窗口。

## 设计原则

- 复杂控件优先语义化 backend，OCR 和坐标点击只作兜底。
- 不新增平行桌面 DSL；优先扩展现有 `desktop_element`、`desktop_capture`、`desktop_vision` 和 `desktop_assert`。
- 输出大对象时文件优先、变量摘要优先，避免大表格或树结构污染 AI 上下文。
- AI 终端先读 `inspect_desktop.capability_matrix`，缺能力时必须改用探测、人工确认或返回不可执行说明。
- 每个高级能力必须有 schema 负例、临时夹具正例、失败证据和文档同步。

## 验收标准

- 新增 action type 和字段风格与现有 `desktop_element`、`desktop_capture`、`desktop_vision` 一致。
- 复杂控件输出写入 `output/desktop-elements/`，大对象不直接塞进变量。
- 相对截图 payload 包含 `source_bounds` 和 `coordinate_space`；相对视觉 payload 同时包含 `source_bounds`、`local_bounds/local_point` 和屏幕全局 `bounds/point`。
- OCR 缺依赖或缺语言包时返回 skipped/能力缺失，不误报框架失败。
- AI 终端不会在能力矩阵缺失时生成高级桌面 action。
- Windows 临时夹具覆盖表格、树、菜单、滚动容器、窗口/控件截图和窗口/控件相对视觉；OCR 基础路径后续补齐。
- `release-matrix` 可按 `--only desktop_advanced_components` 或等价步骤定位该能力。

## 风险

- 不同 UI 框架暴露的控件 pattern 差异很大，临时夹具通过不代表所有业务软件可控。
- 大表格、大树会产生大量数据，必须限制输出并提供分页/截断诊断。
- 菜单窗口短生命周期和焦点变化容易造成 flake，需要等待、截图和失败证据。
- OCR 依赖本机二进制和语言包，安装、路径和字符集会带来环境差异。
- 多显示器、DPI、RDP 缩放和 Retina 坐标换算容易导致点击偏移，需要单独回归。
- macOS AX 复杂控件映射必须等真机补齐，Windows 上不能假验证。
