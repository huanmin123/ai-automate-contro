# 计划-0008-桌面场景任务自动化落地

## 背景

桌面执行线已经具备窗口、控件、键鼠、截图、视觉、OCR、复杂控件、真实 App 和 AI 终端闭环基础能力。下一步需要支持更接近用户目标的连续场景，例如游戏日常、桌面聊天软件定时消息、系统客户端批处理和长期挂机监控。

这类任务不能把所有窗口复查、重试、防卡死和失败恢复都交给 plan。否则 plan 会变成低层控制脚本，AI 生成难度高，维护成本也高。场景任务需要在 runtime 层沉淀通用状态机和守卫策略，plan 只保留业务目标和必要约束。

## 目标

- 明确桌面场景任务的能力边界、风险边界和 plan/runtime 分工。
- 用 mock 游戏、微信式聊天、QQ式定时群消息覆盖场景状态机、防卡死和恢复策略。
- 保持 handbook 干净，不把场景设计、迭代计划和开发验证噪音写入 AI-facing action 文档。
- 将 `desktop-scenarios` 和 `desktop-scenario-apps` 纳入确定性回归矩阵，作为场景层轻量门禁和受控真实窗口门禁。
- 为后续把场景状态机接入真实桌面 action 打基础。

## 范围

本阶段包含：

- 新增 `docs/functions/桌面场景任务自动化设计.md`。
- 新增 `python .\cplan.py self-check desktop-scenarios`。
- 新增 `python .\cplan.py self-check desktop-scenario-apps`。
- mock 覆盖游戏日常、聊天联系人消息、定时群消息。
- mock 覆盖窗口被挡、最小化、启动后窗口未出现、弹窗、重复观察和防无限循环。
- 受控窗口覆盖 mock chat 发送、mock game 日常流程和 mock recovery 最小化/弹窗恢复流程。
- 更新运行说明、测试说明、功能索引和计划索引。
- 将 `desktop_scenarios` 和 `desktop_scenario_apps` 加入 `release-matrix` 默认确定性步骤。

本阶段不包含：

- 控制真实微信、QQ 或真实游戏账号。
- 新增桌面专属定时 DSL。
- 新增普通 plan 的混合 browser/desktop 执行线。
- 绕过验证码、人机验证、反作弊、安全控件、UAC 或 macOS 隐私授权。

## 实施步骤

1. 文档设计
   - 新增场景任务设计文档。
   - 在跨平台电脑控制设计中补充典型场景分类。
   - 在调度设计中说明桌面定时任务复用 `trigger` 和 `cplan schedule`。
   - 更新 docs 索引和计划索引。

2. Mock 场景回归
   - 新增 `desktop_scenarios_check.py`。
   - 定义统一 mock 运行循环：observe、guard、decide、act、verify、recover、stop。
   - 覆盖 `mock_game_daily`、`mock_wechat_blessing`、`mock_qq_schedule`。
   - 输出每个场景的 trace、events、guard_counts、assertions 和 budgets。

3. 受控真实窗口场景回归
   - 新增 `desktop_scenario_apps_check.py`。
   - Windows 上动态生成 WinForms mock chat、mock game 和 mock recovery。
   - 用完整 desktop plan 验证窗口启动、等待、聚焦、observe、控件 dump、写值、原生触发、状态断言、窗口截图、业务结果文件、关闭链路、最小化窗口恢复、阻塞弹窗处理和 `interaction_guard` 证据。
   - 非 Windows 或缺 PowerShell 时返回 `skipped`，不控制真实微信、QQ 或游戏账号。

4. CLI 与矩阵
   - 在 `cplan self-check` 注册 `desktop-scenarios` 和 `desktop-scenario-apps`。
   - 在 CLI 边界自检中加入新命令。
   - 在 release matrix 中加入 `desktop_scenarios` 和 `desktop_scenario_apps` step。

5. 后续真实 runtime 增强
   - 把 mock 中的预算、重复观察和恢复事件结构抽象到 desktop runtime。
   - 将窗口守卫、候选重解析、Windows z-order/topmost 遮挡恢复、观察不变恢复和失败诊断形成统一 payload。
   - 继续增加受控真实窗口场景夹具，例如聊天搜索多匹配、奖励按钮消失、登录过期和网络异常。
   - 真实第三方软件验证保持显式手动触发，不进入默认矩阵。

## 风险

- 真实聊天软件 UI 变化频繁，控件树和 OCR 都可能不稳定，需要先用受控 mock 和窗口夹具验证策略。
- 游戏自绘 UI 依赖图像模板和 OCR，容易受缩放、主题、分辨率和动画影响。
- 定时消息有误发风险，必须要求收件人验证、白名单和必要人工确认。
- 用户在自动化期间移动鼠标、切换窗口、最小化目标窗口或让其它 topmost 窗口遮挡目标会干扰执行，底层必须优先恢复/置前目标窗口并设置重试上限。
- 真实键鼠回归之间会争抢前台窗口和鼠标焦点，不能把 `desktop-components --require-input`、`desktop-scenario-apps`、`desktop-real-app` 等 GUI 回归并行运行。
- 无限循环风险必须由 runtime budget 控制，不能依赖 AI plan 自觉写退出条件。

## 验收标准

- `python .\cplan.py self-check desktop-scenarios` 通过。
- `python .\cplan.py self-check desktop-scenario-apps` 通过。
- `python .\cplan.py self-check desktop-components --require-input` 通过，覆盖真实鼠标/键盘、右键菜单、滚动、双击、右键、滚轮、拖拽和遮挡恢复。
- `python .\cplan.py self-check release-matrix --only desktop_scenarios` 通过。
- `python .\cplan.py self-check release-matrix --only desktop_scenario_apps` 通过。
- `python .\cplan.py self-check cli` 覆盖新 self-check 命令。
- `python .\cplan.py self-check handbook` 通过，确认 handbook 没混入开发计划和回归记录。
- `python -m compileall -q src main.py cplan.py` 通过。
- `git diff --check` 通过。
