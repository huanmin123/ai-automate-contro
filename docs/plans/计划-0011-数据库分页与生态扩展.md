# 计划-0011-数据库分页与生态扩展

## 背景

`sql` 和 `redis` common action 已经具备基础连接、SQLite local-first、事务、批量写入、JSONL 落盘和默认数据库组件自检。下一阶段需要把数据库能力继续推进到更接近真实自动化场景：大表分页读取、稳定的大结果落盘、更多数据库生态候选，以及 AI 生成 plan 时更清楚的选择策略。

这仍然不是第三条执行线。数据库能力继续作为 common action，被 `automation_type: "browser"` 和 `automation_type: "desktop"` 共享。

## 目标

- 为 `sql.query`/`sql.scalar` 增加受控分页字段，减少用户手写不同数据库分页语法的成本。
- 继续强化大结果处理：默认变量池保护，推荐 `rows_path` 和 JSONL，必要时再引入流式写入。
- 明确后续数据库生态扩展顺序：SQL Server、MongoDB、ClickHouse、Elasticsearch 专项能力；DuckDB 已在后续计划中作为本地分析路径接入。
- 让 AI 在生成 plan 时优先使用 SQLite；服务型数据库仅在用户明确提供连接或业务要求时使用。
- 保持真实服务回归为显式可选，不纳入默认离线矩阵。

## 范围

- `sql.query`/`sql.scalar` 新增 `limit`、`offset`、`page_size`、`page` 字段。
- SQLite 自检覆盖分页语义。
- handbook、功能文档和测试说明同步分页字段。
- 创建后续生态扩展路线，但不在本阶段一次性接入所有数据库。

## 不做

- 不新增 `automation_type: "database"`。
- 不把 MongoDB、Elasticsearch、Redis 和 SQL 强行合并为一个通用 action。
- 不默认连接用户的真实数据库服务。
- 不内置 ORM、迁移系统、连接池或数据库服务启动器。

## 设计方向

### 1. SQL 分页字段

推荐写法：

```json
{
  "action": "sql",
  "type": "query",
  "connection": "crm_pg",
  "sql": "select id, status from orders order by id",
  "page_size": 100,
  "page": 2,
  "rows_path": "orders-page-2.jsonl"
}
```

字段语义：

- `limit`: 直接限制返回行数。
- `offset`: 跳过前 N 行。
- `page_size`: 每页行数。
- `page`: 1 基页码，默认 1；和 `page_size` 一起使用。

分页只负责添加受控数据库分页语句，不替用户推断排序条件。大表分页应由用户或 AI 明确 `order by`，保证翻页稳定。

### 2. 大结果处理

- `max_rows` 继续作为变量池保护，默认 1000。
- 指定 `rows_path` 时默认不把完整 rows 放入变量。
- `.jsonl` 继续作为大结果推荐格式。
- 后续如需要超过单页的大结果导出，再补 `stream: true` 或游标式导出能力。

### 3. 数据库生态扩展顺序

优先级建议：

1. SQL Server：企业内网和 Windows 自动化常见，但需要评估 `pyodbc`/ODBC Driver 安装成本。
2. DuckDB：和 SQLite 一样适合本地化数据分析，适合 CSV/Parquet 中间数据处理；已在 [计划-0014-SQL复制同步与DuckDB本地分析落地](./计划-0014-SQL复制同步与DuckDB本地分析落地.md) 中接入。
3. MongoDB：数据模型不同，应新增 `mongo` common action。
4. ClickHouse：适合分析型大表，只在用户需求明确后接入。
5. Elasticsearch：当前由 `http` action 覆盖；需要 bulk index、scroll/search_after 或 mapping 管理时再新增专项 action。

## 实施步骤

1. 为 `sql.query`/`sql.scalar` 增加分页参数解析与跨数据库 SQL 拼接。
2. 增加字段校验，限制 `limit`、`offset`、`page_size`、`page` 的类型和组合。
3. 扩展 SQLite database-components 自检，验证分页结果和输出文件。
4. 更新 `handbook/actions/common/io/sql.md` 和数据库功能设计文档。
5. 后续再按需求接入 SQL Server、MongoDB 或 Elasticsearch 专项 action；DuckDB 已作为 SQL 后端接入。

## 风险

- 没有稳定 `order by` 的分页结果可能在数据库变动时不稳定。
- Oracle、MySQL、PostgreSQL、SQLite 的分页语法不同，需要持续通过真实服务回归验证。
- `limit` 大于 `max_rows` 时仍会触发变量池保护，用户需要显式提高 `max_rows` 或使用 `rows_path` 分页落盘。

## 验收标准

- `sql.query` 支持 `limit`、`offset`、`page_size`、`page`。
- SQLite 自检能验证分页输出。
- `python .\cplan.py self-check database-components` 通过。
- `python .\cplan.py self-check release-matrix --only database_components,compileall,handbook,workspace_clean --fail-fast` 通过。
- handbook 和功能文档清楚说明分页字段与大结果建议。
