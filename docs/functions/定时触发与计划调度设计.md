# 定时触发与计划调度设计

## 背景

当前 plan 是线性执行模型：`steps` 从上到下执行，结束后关闭浏览器和运行资源。这个模型适合一次性自动化，但不适合两类时间需求：

- 运行期触发：浏览器已经登录后，每隔一段时间在同一个会话里执行一组动作或一个子计划。
- 长期调度：每天、每隔一段时间从 CLI 启动一个完整 plan run。

这两类能力必须分开。运行期触发属于单次 plan run 内部的父级控制流；长期调度属于 `cplan` 管理层，负责按时间启动完整 plan。

## 设计原则

- plan 中放入 `steps` 的都是可执行节点，不再引入顶层 `routines` 或 `triggers` 定义区。
- 运行期触发器像 `foreach`、`retry` 一样是父级 action，周期执行体是它的子区域。
- 运行期触发器只在当前 run 和当前浏览器会话内生效，plan 结束时自动失效。
- 长期调度只负责启动 plan，不侵入 plan 语法和浏览器运行态。
- 默认禁止无限运行；没有 `max_runs` 或 `duration_seconds` 时，必须显式 `allow_infinite: true`。
- 触发器执行使用单线程同步语义，不在后台线程里并发操作同一个 Playwright 页面。
- CLI 和 AI 终端共用同一套 schedule 核心模块，不维护两套配置规则。

## Plan 内运行期触发器

`trigger` 直接写在主 `steps` 中。执行器运行到这个节点时进入周期循环，结束后继续父 plan 的后续步骤。

### 内联步骤

```json
{
  "action": "trigger",
  "type": "interval",
  "name": "refresh_every_5_min",
  "every_seconds": 300,
  "run_immediately": true,
  "max_runs": 12,
  "overlap": "skip",
  "on_error": "fail_plan",
  "steps": [
    {
      "action": "navigate",
      "type": "refresh",
      "browser": "main"
    }
  ],
  "save_as": "refresh_status"
}
```

### 周期执行子计划

```json
{
  "action": "trigger",
  "type": "interval",
  "name": "check_accounts",
  "every_seconds": 300,
  "run_immediately": true,
  "max_runs": 12,
  "path": "sub-plans/check-accounts-plan.json",
  "save_as": "check_accounts_status"
}
```

`trigger.path` 使用现有子计划边界：只能引用当前 plan 包内的 `sub-plans/*-plan.json`，不能引用另一个主 `plan.json`。

### 字段

- `name`: 触发器名称，用于日志和状态；不填时使用当前 step 的 `name` 或 `trigger`。
- `type`: 固定支持 `interval`。
- `every_seconds`: 间隔秒数，必须大于 0。
- `steps`: 内联周期执行步骤，与 `path` 二选一。
- `path`: 周期执行子计划，与 `steps` 二选一。
- `run_immediately`: 启动后是否立即执行第一次，默认 `false`。
- `max_runs`: 最大执行次数。
- `duration_seconds`: 最长运行秒数。
- `allow_infinite`: 显式允许无限运行，默认 `false`。
- `stop_condition`: 每轮执行前检查的条件对象，满足时结束。
- `overlap`: 执行体耗时超过间隔时的处理，支持 `skip`、`queue`、`fail`，默认 `skip`。
- `on_error`: 执行体失败时的处理，支持 `fail_plan`、`stop_trigger`，默认 `fail_plan`。
- `save_as`: 触发器结束后把状态对象保存到变量。

### 运行状态变量

每次执行体运行前注入这些变量：

- `trigger_name`
- `trigger_run_index`
- `trigger_started_at`
- `trigger_last_run_at`
- `trigger_next_run_at`

这些变量是当前 run 的临时变量，后续触发器执行会覆盖同名值。内联 `steps` 和 `trigger.path` 指向的子计划都可以使用它们。

## CLI 长期定时计划

长期调度由 `cplan schedule` 管理，配置文件位于当前运行根：

```text
schedules.json
.keygen/schedules-state.json
```

- `schedules.json`: 用户可读、可编辑的调度配置。
- `.keygen/schedules-state.json`: 最近运行、下次运行、错误等本地状态，不提交。
- 删除最后一个 schedule 后，空的调度配置和空状态文件会自动清理。

示例：

```json
{
  "version": 1,
  "schedules": [
    {
      "id": "vsllm_daily",
      "enabled": true,
      "plan_file": "D:\\my-plans\\vsllm-daily-checkin-card\\plan.json",
      "project_root": "D:\\my-plans",
      "timezone": "Asia/Shanghai",
      "trigger": {
        "type": "daily",
        "at": "09:05"
      },
      "concurrency": "skip",
      "timeout_seconds": 1800
    }
  ]
}
```

### CLI 命令

```powershell
python .\cplan.py schedule list
python .\cplan.py schedule add --id vsllm_daily --file D:\my-plans\vsllm-daily-checkin-card\plan.json --daily-at 09:05
python .\cplan.py schedule add --id poller --file .\plans\demo\plan.json --every-seconds 300 --run-immediately
python .\cplan.py schedule enable vsllm_daily
python .\cplan.py schedule disable vsllm_daily
python .\cplan.py schedule remove vsllm_daily
python .\cplan.py schedule run-now vsllm_daily
python .\cplan.py schedule daemon
```

`daemon` 是常驻进程；如果需要系统级唤醒，后续再接 Windows Task Scheduler。初版不默认安装系统任务。

桌面场景任务同样复用这套调度模型。微信/QQ 式定时消息、游戏每日任务和桌面客户端周期检查都应由 `cplan schedule` 启动完整 desktop plan，或在单次 run 内用通用 `trigger` 循环；不新增桌面专属 scheduler 或平行 DSL。窗口探测、登录态确认、收件人验证、视觉识别和失败证据仍由 desktop plan 自己完成。

### AI 终端工具

AI 终端通过结构化工具管理同一份 schedule：

- `list_schedules`
- `add_schedule`
- `remove_schedule`
- `enable_schedule`
- `disable_schedule`
- `run_schedule_now`

AI 只生成配置草案并调用工具写入，不绕过 `cplan` 的校验和运行边界。

## 完成标准

- `trigger` action 作为父级控制流节点，支持 `interval`、`steps` 和 `path`。
- 顶层 `routines` 和 `triggers` 不再是有效 plan 结构。
- `trigger.path` 使用现有 `sub-plans/*-plan.json` 边界和循环引用校验。
- `cplan schedule` 支持配置管理、立即运行和 daemon 单次/常驻检查。
- AI 终端工具注册表包含 schedule 管理工具，且提示词不再生成旧 trigger 语法。
- handbook、docs、测试 plan 和 self-check 都覆盖新能力。
- 真实命令验证通过：plan 校验、trigger 示例运行、schedule add/list/run-now/daemon once、工具注册检查。
