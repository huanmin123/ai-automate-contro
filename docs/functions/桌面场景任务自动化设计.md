# 桌面场景任务自动化设计

## 定位

桌面场景任务自动化面向“像用户一样操作电脑”的连续任务，例如游戏日常、桌面聊天软件定时消息、办公客户端批处理、系统文件对话框和长期挂机监控。它不是新的执行线，也不是新的 plan DSL，而是 `automation_type: "desktop"`、桌面 action、通用控制流、长期 schedule、运行证据和失败恢复策略的组合方案。

核心目标是让 plan 保持简单：plan 描述任务目标、目标 App、目标对象、内容、约束和必要证据；底层 runtime 负责窗口复查、定位重试、候选重解析、遮挡/最小化恢复、防死循环、失败现场和诊断建议。只有底层无法可靠推断的业务选择，才进入 plan。

## 能力结论

在真实登录桌面会话、目标 UI 可见、权限和依赖满足、目标 App 不主动阻止自动化的前提下，可以覆盖下列能力：

- 打开、聚焦、恢复、关闭目标 App 和窗口。
- 控制聊天、办公、系统工具、文件对话框和普通客户端。
- 对控件树可见的 UI 执行读取、点击、输入、选择、菜单、表格、树和滚动。
- 对自绘 UI、游戏画面、图片按钮和 Canvas 使用截图、图像模板、OCR 和候选点兜底。
- 用 `trigger` 做单次 run 内循环，用 `cplan schedule` 做长期定时启动。
- 在关键风险点用 `manual_confirm` 交给用户确认后继续。

不承诺自动完成：

- 锁屏、未登录、无 GUI、RDP 断开、Windows 安全桌面、UAC 提权窗口和跨用户会话。
- macOS 未授权 Accessibility、Screen Recording 或 Automation 的场景。
- 验证码、人机验证、支付、真实下单、真实删除、生产破坏性动作和高风险账号操作。
- 反作弊游戏、银行安全控件、DRM/受保护内容、安全软件刻意拦截的界面。
- 控件树不可见且视觉特征频繁变化、导致无法稳定识别的 UI。

## Plan 与 Runtime 分工

### Plan 应描述

- `automation_type: "desktop"`。
- 目标 App、窗口查询、关键控件 locator、视觉模板或 OCR 关键词。
- 任务目标，例如“给 Alice 和 Bob 发送同一段祝福”“领取每日奖励并完成一次副本”。
- 数据输入，例如联系人列表、消息文本、游戏任务次数、保存路径。
- 必要的安全约束，例如发送前必须确认收件人、只运行一次、不处理支付弹窗。
- 必要的证据步骤，例如 `desktop_capture type=observe`、窗口列表、控件 dump、截图、断言。
- 需要人工判断的断点，例如首次登录、验证码、权限授权、发送前人工确认。

### Runtime 应处理

- 每个会改变桌面的 action 前确认目标窗口仍存在、可见且处于前台。
- 目标窗口被覆盖、失焦、最小化或刚启动未出现时，自动聚焦、恢复、等待或重连。
- 鼠标坐标动作前复查坐标是否仍位于目标窗口和可点击屏幕范围内。
- 候选点、控件 locator 和视觉命中在执行前重新解析，避免使用过期坐标。
- 观察结果长时间不变时执行受限恢复动作，超过预算后失败，不无限循环。
- 弹窗、系统对话框、权限不足、依赖缺失和识别失败时输出结构化诊断。
- 失败时保存窗口列表、活动窗口、截图、控件诊断、候选点、坐标信息和能力矩阵。

### 必须放进 Plan 的情况

- 业务目标有多种合法选择，例如选哪个联系人、哪个副本、哪个群、哪个文件。
- 需要用户确认风险，例如真实发送消息、真实交易、删除数据、账号安全提示。
- 需要用户提供稳定视觉模板、OCR 文本、业务数据或白名单。
- Runtime 无法判断某个弹窗是可关闭提示还是必须处理的业务弹窗。

## 场景任务运行模型

场景任务按下面循环执行，循环预算由 runtime 控制：

```text
observe desktop
  -> guard target window
  -> resolve target candidate
  -> execute next planned action
  -> verify progress
  -> recover if blocked
  -> stop on goal, manual_confirm, failure, or budget
```

运行状态最少包含：

- `target_app`: 目标 App 或进程。
- `target_window`: 当前目标窗口候选。
- `surface`: 当前任务界面，例如聊天列表、聊天窗口、游戏主页、副本战斗。
- `active_target`: 当前联系人、群、副本或文件。
- `last_observation`: 最近一次观察摘要。
- `progress_markers`: 已完成的联系人、已刷次数、已领取奖励、已保存文件。
- `guard_counts`: 聚焦、恢复、弹窗关闭、重复观察恢复等计数。
- `budgets`: 最大步骤数、最大重复观察次数、最大恢复次数、最大单动作重试次数。

当前可复用场景运行结构在 `src/ai_automate_contro/engine/desktop/scenarios.py`，用于纯 mock 场景回归和后续真实 runtime 复用。它负责场景上下文、观察记录、guard 动作、恢复计数、事件记录、重复观察预算和总恢复预算；plan 仍只描述任务动作和必要证据。

停止条件必须明确：

- 目标完成。
- 触发人工确认。
- 达到最大步骤、最大恢复次数或单动作重试上限。
- 目标窗口消失且无法恢复。
- 关键识别证据缺失。
- 出现高风险弹窗或权限边界。

## 窗口与遮挡策略

桌面自动化的优先级高于用户临时点击，但不能破坏系统安全边界。Runtime 对所有真实输入和控件写入执行目标窗口守卫：

- `missing`: 等待或重新启动目标 App；仍失败则保存失败现场。
- `minimized`: 还原窗口并重新读取活动窗口。
- `covered` 或 `background`: 聚焦目标窗口，短暂等待后验证活动窗口。
- `covered by topmost`: Windows native backend 会在交互前把目标窗口提升到 topmost 栈前方，避免坐标输入被其它 topmost 窗口截获。
- `active mismatch`: 不执行键鼠，先重新聚焦或失败。
- `outside window`: 坐标动作默认失败，除非 step 显式允许窗口外点击。
- `permission denied`: 返回权限诊断，必要时让用户授权或人工确认。

现有实现已对真实键鼠和会改变桌面的 `desktop_element` action 增加交互窗口守卫、Windows z-order 提升和有限重试。`desktop_capture type=observe`、窗口/控件截图和受控场景回归会记录 `interaction_guard` 证据。后续场景层应继续复用这套守卫，避免 plan 手写“先检查窗口、再聚焦、再重试”的重复步骤。

## 识别定位策略

优先级从高到低：

1. App 自身 API、配置、导入导出能力。
2. 控件树语义定位，例如 UIA/AX 的 name、automation id、role、value、selection。
3. 窗口查询、菜单路径、快捷键和剪贴板。
4. `desktop_capture type=observe` 生成的候选点和控件摘要。
5. `desktop_vision type=locate_image` 图像模板。
6. `desktop_vision type=locate_text` OCR 文本。
7. 坐标点击、拖拽和滚动兜底。

视觉和坐标只能作为兜底。最终点击前必须确认候选点仍在目标窗口内，且 `screen_clickable=true`。离线图片、低置信候选和 `screen_clickable=false` 只能作为证据，不能直接点击。

## 游戏日常场景

适用范围：

- 自有游戏、内部测试环境、无反作弊阻断、用户明确授权的测试任务。
- 签到、领取奖励、进入固定副本、简单战斗循环、领取副本奖励。

推荐计划：

- 首先用 `inspect_desktop` 或 `desktop_capture type=observe` 确认窗口、截图和能力矩阵。
- 用控件树时优先 `desktop_element`，自绘 UI 使用图片模板和 OCR。
- 将副本次数、任务名称、按钮模板、完成 OCR 文本写成变量。
- 每轮战斗必须有进度证据，例如画面变化、奖励弹窗、计数变化、OCR 文本变化。
- 观察重复超过阈值后由 runtime 执行有限恢复，例如重新聚焦、ESC、关闭弹窗、回到主页。
- 仍无法确认进度时失败并保存截图，不继续盲点。

不应默认做：

- 绕过反作弊、验证码、人机验证。
- 自动充值、交易、分解装备、删除角色或影响生产资产。
- 长时间无限刷图。必须有次数、时长或 `allow_infinite` 这类显式边界。

## 桌面聊天软件场景

适用范围：

- 已登录的本机桌面聊天客户端。
- 用户明确提供收件人、群、消息内容和发送时间。
- 测试或低风险消息。真实大量群发必须加人工确认或更严格白名单。

推荐计划：

- 先确认目标窗口和账号状态。
- 对联系人或群使用搜索框、控件 locator、OCR 或图像模板定位。
- 输入消息后，发送前再次读取当前聊天标题或 OCR 识别收件人。
- 每个收件人只发送一次，运行状态记录 `sent_messages`。
- 发送后保存截图、草稿状态或控件文本证据。
- 弹出同步提示、更新提示、登录过期、网络异常时由 runtime 分类处理；无法分类时人工确认。

发送消息的安全门禁：

- 计划必须包含收件人或群列表。
- 计划必须包含消息正文或正文来源。
- 发送前必须有当前收件人验证。
- 不明确的收件人、搜索结果多匹配、账号切换提示、登录过期和安全验证必须停止或人工确认。

## 调度组合

桌面场景不新增专属定时 DSL：

- 长期定时启动完整任务用 `cplan schedule`。
- 单次 run 内周期检查用通用 `trigger`。
- schedule 只负责启动 plan；真正的窗口探测、登录态确认、任务执行和失败证据仍在 desktop plan 内完成。

示例组合：

```powershell
python .\cplan.py schedule add --id daily_blessing --file D:\plans\daily-blessing\plan.json --daily-at 08:30
python .\cplan.py schedule run-now daily_blessing
python .\cplan.py schedule daemon --once
```

## Mock 回归策略

默认回归不能依赖真实微信、QQ、游戏账号或外部服务器。场景层用 mock 验证任务状态机和恢复策略，用组件回归验证真实桌面输入和窗口控制：

- `desktop-scenarios`: 纯 Python mock，验证游戏日常、聊天联系人消息、定时群消息、窗口恢复、弹窗关闭、重复观察防卡死、收件人确认和边界预算。
- `desktop-scenario-apps`: 受控真实窗口场景回归，Windows 上动态生成 WinForms mock chat、mock game 和 mock recovery，用完整 desktop plan 验证窗口启动、等待、聚焦、observe、控件 dump、写值、原生触发、状态断言、窗口截图、业务结果文件、关闭链路、最小化窗口恢复、阻塞弹窗处理和 `interaction_guard` 证据；非 Windows 或缺 PowerShell 时返回 `skipped`。
- `desktop-components`: 受控 WinForms/WPF/Tkinter 窗口，验证真实控件、窗口、输入、截图、视觉和 OCR 能力。
- `desktop-real-app`: 显式真实 App 回归，验证系统窗口、终端、Explorer/TextEdit 和文件对话框。
- 真实微信、QQ、真实游戏只作为用户显式发起的本地验证，不进入默认 release matrix。

运行命令：

```powershell
python .\cplan.py self-check desktop-scenarios
python .\cplan.py self-check desktop-scenario-apps
python .\cplan.py self-check release-matrix --only desktop_scenarios
python .\cplan.py self-check release-matrix --only desktop_scenario_apps
```

## 后续增强

- 将窗口守卫结果、候选点重解析、z-order 遮挡恢复和失败恢复继续统一写入 action payload。
- 继续丰富 AI 终端场景模板提示，让 AI 在更多业务类型中先确认执行线、风险、收件人/目标和人工确认策略，再写 plan。
- 继续增加受控窗口异常场景，例如聊天搜索结果多匹配、游戏奖励按钮消失、网络断开和登录过期。
