### 组件合并原则

同族能力按参数结构收敛：

- `navigate`: 页面跳转、刷新、前进、后退
- `page`: 新开页面、切换页面、关闭页面
- `element`: 点击、悬停、填充、清空、选择、上传等元素操作
- `wait`: 时间、选择器、URL、文本、数量等待
- `extract`: 文本、值、属性、HTML、数量、表格提取
- `assert`: 选择器、文本、值、URL、数量断言
- `capture`: 截图、HTML、storage state 保存
- `read`: 通过 `type` 读取 `json`、`text`、`csv`、`excel`
- `write`: 通过 `type` 写出 `json`、`text`、`csv`、`excel`、`variables`
- `table`: 通过 `type=filter/select/sort/dedupe` 处理 Excel、CSV、JSON、SQL 等来源的行数组
- `http`: 通过 `type=request` 发起 HTTP(S) 请求，支持常见方法、表单、本地文件上传和响应落盘
- `sql`: 通过 `type=query/scalar/execute/executemany/bulk_insert/copy/transaction` 访问 SQLite、DuckDB、PostgreSQL、MySQL/MariaDB、Oracle，支持分页查询、事务、批量写入和跨连接复制；SQLite 是本地化优先路径，DuckDB 是按需本地分析路径
- `redis`: 通过 `type=get/set/delete/hget/hset/hgetall/lpush/rpush/lrange/sadd/smembers/expire/command/pipeline` 访问 Redis，pipeline 可分批执行
- `command`: 通过 `type=run` 同步调用受控本地命令，把 stdout/stderr 保存为变量或写入 `output/commands/`
- `desktop_app` / `desktop_*`: 第二执行线，App 启动、窗口、输入、截图、等待和控件操作拆为独立 action，不并入 `command`

合并原则：

- 参数结构一致、只差序列化格式的能力合并到同一个组件。
- 参数生命周期不同的能力单独保留，例如 `open_browser`、`run_sub_plan`、`trigger`、`foreach`、`retry`、`wait_for_popup`、`wait_for_download`、AI 节点。
- 单次 run 内的周期动作使用 `steps` 中的父级 `trigger` action，周期执行体写入 `trigger.steps` 或通过 `trigger.path` 引用子计划；长期按时间启动完整 plan 使用 `cplan schedule`。
- 所有写入动作只能产出到当前 plan 包的 `output/`，并按组件进入固定分区。

## 设计目标

- 对业务站点保持低耦合
- 对流程编排保持高可组合性
- 对单个需求保持清晰边界

## 当前范围边界

当前阶段的已实现运行时代码以浏览器自动化、HTTP 客户端、受控本地命令和桌面 Phase 0 为主。`command` 只覆盖同步命令调用，不扩展桌面窗口、后台服务、系统信息采集和坐标级键鼠输入能力；桌面 App 启动和人类可见桌面操作进入 `automation_type: "desktop"` 执行线。

跨平台电脑控制已经进入第二线 Phase 0：通过顶层 `automation_type`、稳定 plan schema、独立 desktop session 和 Windows/macOS native backend adapter 推进。当前已覆盖 `open_desktop`、`desktop_app launch`、窗口列表/聚焦/关闭/最小化/最大化/还原、`desktop_element` 控件列表/定位/文本/状态/点击/写值/触发、键鼠输入、坐标点击/双击/右键/滚动/拖拽、截图、等待、窗口/截图/控件断言和失败现场。浏览器和桌面 action 必须按执行线隔离，详见 [执行线隔离与Plan类型设计](../architecture/执行线隔离与Plan类型设计.md)、[跨平台电脑控制设计](./跨平台电脑控制设计.md) 和 [计划-0006-跨平台桌面自动化第二线落地](../plans/计划-0006-跨平台桌面自动化第二线落地.md)。

## 最新补强

- 新窗口场景：通过 `wait_for_popup` 捕获 popup，并注册为命名页面，继续纳入统一多页面调度。
- 前后端联动场景：通过 `wait_for_network` + `type: request/response` 验证按钮点击、提交等动作是否真正触发目标接口。
- 接口自动化场景：通过 `http` + `type=request` 发起不依赖浏览器的 GET、POST、PUT、PATCH、DELETE、HEAD、OPTIONS、表单和本地文件上传请求。
- 本地处理场景：通过 `command` + `type=run` 调用当前 plan 包内脚本，并把 JSON stdout 继续传给后续 HTTP、条件分支或文件写出步骤。
- 本地数据库场景：通过 `sql` + SQLite 把抓取数据、桌面读取结果或中间状态写入当前 plan 包 `output/sql/*.db`，不需要部署数据库服务；需要本地分析时可按需使用 DuckDB。
- 服务数据库场景：通过 `sql` 连接 PostgreSQL/MySQL/Oracle 查询、分页读取、更新、批量入库、跨连接复制或事务写入，通过 `redis` 读写缓存、状态和队列。
- 数据表场景：通过 `extract` + `type: table` 直接把页面表格提取为二维数组或字典数组。
- 文件结果沉淀：通过 `write` + `type: csv` 把提取结果、执行中间结果写入 CSV，方便后续回归与人工校验。
- Excel 办公数据场景：通过 `read/write` + `type: excel` 读取人员名单、财务流水和报表模板，通过 `table` 对行数组做筛选、选列、排序和去重，再输出 Excel/CSV/JSON。
- 多页面一致性修复：条件判断、重试等待、休眠等待统一改为基于当前活动页面执行，避免单页模型残留。
