# 跨平台电脑控制设计

## 定位

电脑控制是浏览器自动化之外的第二条执行线。它复用同一套 plan 包、变量、输出和运行报告模型，但不能和浏览器 action 混在同一个普通 plan 里。每个主 `plan.json` 必须通过 `automation_type` 声明自己属于 `browser` 或 `desktop`，执行器再选择对应解析路径。

桌面控制的目标不是“运行终端命令”，也不是只控制某一个指定 App。它面向真实 Windows/macOS 桌面会话，尽量覆盖人像平时操作电脑一样完成的本机操作：打开 App、切换窗口、点击菜单和按钮、输入文本、拖拽、滚动、复制粘贴、保存文件、处理系统弹窗、读取屏幕内容、等待视觉变化，并在需要时通过图像识别和 OCR 定位目标。

当前代码已落地桌面基础执行线、原生轻量 backend、窗口列表/聚焦、输入、截图、快照、等待窗口、`desktop_assert` 和桌面失败现场。本文继续沉淀 Windows/macOS 桌面自动化的目标方案、平台边界、依赖选择和分阶段落地顺序。编码时还必须遵守 [桌面Action与Runtime契约](./桌面Action与Runtime契约.md)，实现变更必须同步更新执行器、校验、手册、测试计划和验证说明。

## 能力愿景

长期目标是在一个已登录、未锁屏、有真实 GUI 和必要系统权限的桌面会话里，让 plan 可以编排下列人类可见、可操作的动作：

- 启动、连接、聚焦、最小化、最大化、移动、关闭 App 和窗口。
- 在窗口、菜单、工具栏、按钮、输入框、列表、表格、树、对话框之间导航。
- 执行键盘输入、快捷键、鼠标移动、单击、双击、右键、拖拽、滚轮和复制粘贴。
- 操作文件打开、保存、上传、下载、选择目录等系统对话框。
- 通过控件树读取文本、状态、选中项、可用性和位置。
- 通过截图、图像定位和 OCR 识别自绘界面、画布、图标、图片按钮和不可访问控件。
- 等待窗口、控件、图像、文字、文件或状态变化。
- 在自动化无法可靠判断或触及系统权限边界时，用 `manual_confirm` 交给用户完成当前步骤后继续。

这里的“像人一样操作电脑”指的是在操作系统允许、当前用户会话可见、目标 UI 可被控件树或视觉识别定位的前提下执行键鼠和窗口操作。它不是无条件突破系统安全机制，也不保证所有人能手工完成的动作都可以稳定自动化。

## 设计原则

- plan schema 保持稳定，平台差异收敛到 backend adapter。
- 浏览器和桌面执行线完全隔离：浏览器 plan 不加载桌面 action，桌面 plan 不加载浏览器 action；两者只共享通用 action、变量、输出和控制流。
- 优先使用语义化、可观测、可回放的控制方式，最后才使用坐标级键鼠输入。
- Windows 和 macOS 都是一等目标；不要把 Windows 专用库设计成通用核心。
- 桌面 action 不并入 `command`。`command` 继续只负责同步本地命令，桌面控制使用独立 session、backend、action 和失败证据。
- 桌面自动化必须保留运行证据：截图、窗口列表、控件快照、定位结果、权限诊断和失败现场都写入当前 plan 包 `output/`。
- AI 为真实桌面应用创建最终 plan 前，必须先获取可执行证据，例如窗口列表、控件树、截图、图像定位结果或人工确认，不能凭文字描述猜坐标。
- 不绕过操作系统权限、UAC、管理员边界、macOS TCC 隐私授权、安全桌面、锁屏或不同用户会话隔离。
- 桌面线以“人类可见桌面会话”为运行前提。无 GUI、锁屏、远程会话断开、权限不足、受保护内容或安全控件场景应返回明确错误或 skipped，不伪装成可自动化。

## 能力分层

按可靠性从高到低排序：

1. 应用自身 API、CLI、配置文件、导入导出接口。
2. 浏览器自动化，继续使用 Playwright。
3. 系统命令和脚本，继续使用已落地的 `command` action。
4. 系统信息和进程管理，后续统一封装为 `system` / `process` action。
5. 桌面窗口和控件级自动化，统一封装为 `open_desktop`、`desktop_window`、`desktop_input`、`desktop_capture`、`desktop_wait`、`desktop_assert` 等 action。
6. 截图、OCR、图像识别和坐标级键鼠输入，只作为兜底方案。

## 能力范围

### 应覆盖

- 通用桌面 App：记事本、TextEdit、Finder、资源管理器、计算器、设置窗口、传统客户端和常见业务软件。
- 办公类 App：Excel、Word、桌面表格、文件打开/保存对话框，优先使用应用 API、脚本接口或控件语义，必要时用键鼠和截图兜底。
- 跨 App 流程：从一个 App 复制内容到另一个 App、下载文件后打开桌面程序处理、保存结果到指定位置。
- 系统级交互：窗口切换、菜单点击、快捷键、剪贴板、屏幕截图、权限确认提示的人工交接。
- 自绘或控件树不足的 UI：通过截图、图像匹配、OCR 和坐标兜底处理。

### 不承诺自动完成

- 锁屏、未登录、无桌面会话、Windows Service Session 0、纯 headless CI、远程桌面断开后的 GUI 操作。
- Windows UAC 安全桌面、管理员完整性级别高于当前进程的窗口、受保护系统界面。
- macOS 未授权 Accessibility、Screen Recording、Automation 或目标 App 拒绝 Apple Events 的场景。
- 验证码、人机验证、短信/邮箱二次验证、真实支付、真实下单、真实删除或生产数据变更等需要人工判断或外部确认的场景。
- DRM/受保护内容、银行安全控件、反作弊游戏、虚拟化隔离桌面或安全软件刻意阻止自动化的场景。
- UI 持续变化、控件树不可见且视觉特征不稳定，导致无法可靠定位目标的场景。

## 总体架构

顶层 plan 类型：

```json
{
  "automation_type": "desktop",
  "variables": {},
  "steps": []
}
```

`automation_type: "desktop"` 时，执行器只加载桌面专属 action 和通用 action。`open_browser`、`navigate`、`element`、浏览器 `mouse`、浏览器 `keyboard`、浏览器 `capture` 等浏览器 action 必须在校验阶段失败。完整隔离规则见 [执行线隔离与Plan类型设计](../architecture/执行线隔离与Plan类型设计.md)。

推荐新增桌面执行层：

```text
src/ai_automate_contro/engine/desktop/
  session.py
  backends/
    base.py
    windows.py
    macos.py
    vision.py
```

`RuntimeState` 增加独立桌面会话，不复用浏览器 `sessions`：

```python
desktop_sessions: dict[str, DesktopSession]
```

`DesktopSession` 保存：

- `name`
- `platform`
- `backend_name`
- `backend`
- `current_window`
- `timeout_ms`
- `metadata`

backend 接口保持小而稳定：

```python
class DesktopBackend:
    def snapshot(self) -> dict: ...
    def screenshot(self, path, region=None) -> dict: ...
    def list_windows(self) -> list[dict]: ...
    def focus_window(self, query: dict) -> dict: ...
    def click(self, target: dict) -> dict: ...
    def type_text(self, text: str, delay_ms: int = 0) -> dict: ...
    def hotkey(self, keys: list[str]) -> dict: ...
    def wait(self, condition: dict, timeout_ms: int) -> dict: ...
    def close(self) -> None: ...
```

## 平台 backend

### Windows

首选技术栈：

- `pywinauto`: Windows 桌面控件自动化主线，优先使用 `uia` backend，传统软件可使用 `win32` backend。
- `pywin32`: 窗口句柄、前台窗口、底层 Win32 API 和 `SendInput` 等补充能力。
- `mss`: 屏幕截图。
- `opencv-python`: 图像模板匹配。
- `pyautogui`: 坐标级鼠标、键盘和图像兜底。
- `pyperclip`: 长文本粘贴和剪贴板辅助。

Windows 优先级：

1. UI Automation / Win32 控件级操作。
2. 窗口句柄、快捷键、菜单和剪贴板。
3. 截图、图像定位、OCR。
4. 坐标点击和拖拽。

需要处理的边界：

- UAC、安全桌面、管理员权限和完整性级别限制。
- RDP 断开、锁屏、Windows Service / Session 0 等无交互桌面场景。
- 高 DPI、多显示器、负坐标和缩放导致的坐标错位。
- 自绘 UI、游戏、Canvas、部分 Electron/Qt 应用控件树不完整。

### macOS

首选技术栈：

- `pyobjc-framework-ApplicationServices` / `AXUIElement`: macOS Accessibility API 控件树和 UI 操作主线。
- AppleScript / System Events: 菜单、快捷键、部分 App GUI scripting 和 Apple Events 控制。
- Quartz / CoreGraphics: 屏幕截图、窗口信息和坐标相关能力。
- `mss`: 屏幕截图兜底。
- `opencv-python`: 图像模板匹配。
- `pyautogui`: 键鼠和坐标兜底。
- `pyperclip`: 文本剪贴板辅助。

macOS 优先级：

1. 应用自身 AppleScript/CLI/API。
2. Accessibility API 控件级操作。
3. System Events 快捷键、菜单和窗口交互。
4. 截图、图像定位、OCR。
5. 坐标点击和拖拽。

macOS 权限边界：

- Accessibility: 模拟键盘、鼠标、读取和控制 UI 需要用户授权。
- Screen Recording: 截图、OCR、图像定位需要用户授权。
- Automation: 通过 Apple Events / AppleScript 控制 Finder、Chrome、Excel 等目标 App 时，系统按目标 App 弹出授权。
- Input Monitoring: 只有录制全局键鼠或监听快捷键时需要，普通自动化执行不默认依赖。

代码可以检测权限、触发系统提示、打开系统设置页并暂停等待用户确认，但不能静默替用户授权，也不能自动点击系统隐私授权。开发阶段不需要 Apple Developer ID；正式分发给普通用户时建议使用固定 `.app`、稳定 bundle id、Developer ID 签名和 notarization，否则 Gatekeeper 和 TCC 授权体验会很差。

macOS 局限：

- 权限绑定到启动主体。从 Terminal、VS Code、打包后的 `.app` 启动，会被 macOS 视为不同授权对象。
- AX 控件树不一定稳定，很多应用只暴露泛化 group/button/text field，缺少类似 Windows `AutomationId` 的稳定字段。
- Retina 下逻辑点和物理像素不同，截图、OpenCV 和鼠标点击必须统一坐标体系。
- GUI 自动化通常要求真实登录桌面会话，锁屏、无头 CI 或远程会话断开时不可靠。

## 不采用的主线方案

### 纯 PyAutoGUI 坐标方案

优点是简单、跨平台、容易快速验证。缺点是稳定性最低，受 DPI、窗口位置、多显示器、主题、语言、动画和遮挡影响。它只适合作为兜底 backend，不适合作为项目核心抽象。

### 纯图像识别方案

图像定位适合自绘 UI、画布、游戏或控件树不可用的场景，但模板对缩放、主题、字体抗锯齿和窗口状态敏感；OCR 对小字号、低对比度和中英文混排也不稳定。它应作为跨平台兜底层，而不是第一定位方式。

### WinAppDriver / Appium 作为第一实现

WinAppDriver 生态维护状态不适合作为第一主线。Appium Windows Driver 更适合团队已有 Appium 资产或远程执行场景，当前项目更适合先用本地 Python backend 保持执行器简单。

### Robot Framework / RPA Framework 直接嵌入

这些框架适合完整 RPA 工程，但会引入另一套 DSL、日志、执行模型和生态约束。当前项目已有 JSON plan 执行器，应复用现有 action、变量、输出、失败采集和 AI 终端调试模型。

### FlaUI / .NET sidecar 作为第一实现

FlaUI 的 Windows UIA 能力强，可以作为后续增强路线。首期直接引入 .NET sidecar 会增加构建、进程通信和分发复杂度；只有当 pywinauto 在 UIA pattern、性能或兼容性上明显不足时再引入。

## 输出证据

桌面运行产物仍必须位于当前 plan 包 `output/`：

- `output/desktop-screenshots/`: 桌面截图、窗口截图、区域截图。
- `output/desktop-state/`: backend、平台、权限、显示器、DPI、活动窗口、当前焦点等快照。
- `output/desktop-windows/`: 窗口列表、窗口匹配结果。
- `output/desktop-elements/`: 控件树、候选控件、selector 匹配结果。
- `output/desktop-vision/`: 图像匹配、OCR 原图、裁剪图、标注图、原始识别文本。
- `output/failure-desktop-screenshots/`: 失败时桌面截图。
- `output/failure-desktop-state/`: 失败时桌面状态、窗口列表、权限诊断和目标定位配置。

失败采集应在浏览器失败现场之外，遍历 `state.desktop_sessions`，记录桌面 backend、平台权限、活动窗口、窗口列表、截图和当前 action 的 target 摘要。

## 依赖拆分草案

```toml
[project.optional-dependencies]
desktop = [
  "pyautogui>=0.9.54,<1.0.0",
  "pyperclip>=1.9.0,<2.0.0",
  "mss>=10.0.0,<11.0.0",
  "pywinauto>=0.6.8,<1.0.0; sys_platform == 'win32'"
]
```

Phase 0 只落地 `desktop` 可选依赖组。`pywinauto`、`mss`、OpenCV、OCR 和 pyobjc 相关能力按后续阶段接入；`PyDirectInput` 仅适合 Windows 下 DirectInput 特殊场景，例如部分游戏或普通输入库不生效的应用。它不进入默认依赖，也不作为默认桌面控制方案。

## 分阶段落地

### 第一阶段：Windows/macOS 基础桌面线（已落地）

- 按 [桌面Action与Runtime契约](./桌面Action与Runtime契约.md) 固定 action schema、runtime payload、输出分区、失败采集和 self-check。
- 在 plan validator 中新增 `automation_type` 必填校验，拒绝跨线 action。
- 在 handbook 中建立 `browser/` 和 `desktop/` 双入口，桌面 action 文档不放进浏览器 action 目录。
- 新增 `DesktopSession` 和 backend adapter。
- 实现 `open_desktop`、`close_desktop`。
- 实现 `desktop_window type=list/focus`。
- 实现 `desktop_input type=type_text/hotkey`。
- 实现 `desktop_capture type=screenshot/snapshot`。
- 实现 `desktop_wait type=window`。
- 实现 `desktop_assert type=window/screenshot`。
- Windows Phase 0 backend 使用 `ctypes` 做窗口枚举/聚焦，截图使用 Pillow `ImageGrab`，输入通过可选 `pyautogui`。
- macOS Phase 0 backend 使用 AppleScript 做基础窗口查询，截图使用 Pillow `ImageGrab`，输入通过可选 `pyautogui` 并触发系统权限。
- 失败采集写入桌面截图和状态 JSON。

### 第二阶段：控件级定位和断言

- Windows 支持 `uia` / `win32` 控件 selector。
- macOS 支持 AX role、title、description、value、identifier 等字段。
- 扩展 `desktop_assert` 到控件级断言。
- 新增控件级 `click`、`set_text`、`get_text`、`wait`。
- 增加控件树 dump 和候选匹配报告。

### 第三阶段：视觉兜底

- 接入 OpenCV 模板匹配。
- 接入 OCR locator。
- 支持区域截图、活动窗口截图、多显示器坐标和高 DPI/Retina 坐标统一。
- 图像/OCR 失败时保存原图、裁剪图、标注图、候选分数和原始 OCR 文本。

### 第四阶段：高级 backend

- 评估 macOS 更完整的 AXUIElement 封装。
- 评估 Windows FlaUI sidecar。
- 评估 Appium/WinAppDriver/NovaWindows 作为可选 provider。
- 增加桌面录制、步骤级 trace、权限诊断工具和 AI 终端桌面探测工具。

## 验证路线

新增 `python .\cplan.py self-check desktop-components` 后分三层验证：

1. 纯校验层：不依赖真实 GUI，覆盖 action type、必填字段、输出路径和字段互斥。
2. backend 探测层：检查平台、依赖、权限、是否能截图、是否能列窗口。
3. 真实动作层：Windows 用 Notepad，macOS 用 TextEdit 或系统可用轻量 App，执行聚焦、输入、保存、截图、断言和关闭。

无 GUI、锁屏、权限不足或 CI 环境应返回 `skipped` 和原因，不把平台能力缺失误报为框架失败。
