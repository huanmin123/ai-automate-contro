# 桌面复杂控件与视觉增强设计

## 定位

本文承接 [跨平台电脑控制设计](./跨平台电脑控制设计.md)、[桌面Action与Runtime契约](./桌面Action与Runtime契约.md) 和 [桌面视觉定位设计](./桌面视觉定位设计.md)，用于规划桌面线下一阶段的大能力：复杂控件、窗口/控件相对视觉、OCR 和坐标校准。

Phase 0 已经提供窗口、基础控件、键鼠、截图、图像模板定位、失败证据和 AI 终端桌面闭环。下一阶段不新建第三条执行线，也不把桌面能力并入 `command`；所有增强继续挂在 `automation_type: "desktop"`、`desktop_element`、`desktop_capture`、`desktop_vision`、`desktop_assert` 和 backend adapter 内。

截至 2026-06-28，复杂控件首批已在 Windows native backend 落地 `desktop_element type=get_table/select_cell/get_tree/expand_tree/collapse_tree/select_tree/invoke_menu/scroll_element` 和 `invoke_menu open_context_menu=true`，并由 `desktop-components` 的临时 WinForms 夹具覆盖 DataGridView、TreeView、MenuStrip、ContextMenuStrip、滚动 Panel、输出 JSON 和标注证据。`desktop_capture target=window/element`、`desktop_vision source_target=window/element` 和 `desktop_vision type=locate_text` 已落地；窗口/控件截图和视觉命中已返回屏幕全局 `bounds`、source 局部 `local_bounds`、`coordinate_space` 和 `coordinate_diagnostics`。完整 DPI/Retina/多显示器校准仍按后续 `CoordinateMapper` 推进。

## 优先级

1. 复杂控件语义能力：表格、树、菜单、滚动容器。
2. 窗口/控件相对截图和视觉定位：按 Window Query 或 Element Locator 截图、定位和坐标回填。
3. OCR locator：`desktop_vision type=locate_text`，作为控件树和图像模板不可用时的取证兜底。
4. 多显示器、DPI/Retina 坐标校准：把截图、控件 bounds、鼠标输入和视觉结果统一到同一坐标空间。
5. 更真实的跨 App 样例：在能力稳定后扩展系统设置、复杂资源管理器操作和业务夹具。

复杂控件优先于 OCR。语义控件能直接读取结构、状态和可操作 pattern；OCR 只读屏幕像素，稳定性受语言包、字体、缩放、主题和图像质量影响。

## 复杂控件

### 目标

在不改变 `desktop_element` 总体模型的前提下，扩展现有 Element Locator 和控件 payload，让 plan 能可靠操作常见桌面控件：

- 表格：读取行列、单元格文本、表头、选中行、点击/选择单元格。
- 树：展开、折叠、选择节点、按路径定位节点。
- 菜单：打开菜单、选择菜单项、处理上下文菜单。
- 滚动容器：滚动到目标控件、分页查找、记录可见区域和候选范围。

### Action 扩展

优先扩展 `desktop_element` 的 `type`，不新增平行 action：

```json
{
  "action": "desktop_element",
  "desktop": "desk",
  "type": "get_table",
  "title_contains": "Demo",
  "automation_id": "OrdersGrid",
  "path": "orders-grid.json",
  "save_as": "orders_grid"
}
```

```json
{
  "action": "desktop_element",
  "desktop": "desk",
  "type": "select_cell",
  "title_contains": "Demo",
  "automation_id": "OrdersGrid",
  "row": 2,
  "column": "Status",
  "save_as": "selected_cell"
}
```

```json
{
  "action": "desktop_element",
  "desktop": "desk",
  "type": "expand_tree",
  "title_contains": "Demo",
  "automation_id": "NavigationTree",
  "tree_path": ["Settings", "Accounts"],
  "save_as": "tree_node"
}
```

```json
{
  "action": "desktop_element",
  "desktop": "desk",
  "type": "invoke_menu",
  "title_contains": "Demo",
  "menu_path": ["File", "Save As"],
  "save_as": "menu_result"
}
```

```json
{
  "action": "desktop_element",
  "desktop": "desk",
  "type": "invoke_menu",
  "title_contains": "Demo",
  "automation_id": "OrderList",
  "open_context_menu": true,
  "menu_path": ["Open"],
  "save_as": "context_menu_result"
}
```

建议新增或扩展的 `type`：

| 类型 | 作用 | 输出分区 |
| --- | --- | --- |
| `get_table` | 读取表格结构、行列、表头、可见单元格 | `output/desktop-elements/` |
| `select_cell` | 选择或点击表格单元格 | `output/desktop-elements/` + `desktop-annotations/` |
| `get_tree` | 读取树结构和节点摘要 | `output/desktop-elements/` |
| `expand_tree` | 展开树节点路径 | `output/desktop-elements/` + `desktop-annotations/` |
| `collapse_tree` | 折叠树节点路径 | `output/desktop-elements/` + `desktop-annotations/` |
| `select_tree` | 选择树节点路径 | `output/desktop-elements/` + `desktop-annotations/` |
| `invoke_menu` | 按菜单栏或上下文菜单路径触发菜单项 | `output/desktop-elements/` + `desktop-annotations/` |
| `scroll_element` | 在控件容器内滚动到目标或偏移 | `output/desktop-elements/` + `desktop-annotations/` |

### Locator 扩展

新增字段应保持跨平台命名，平台专属字段只放在 backend payload 的 `platform_details`：

| 字段 | 作用 |
| --- | --- |
| `row` | 表格行索引，0-based |
| `column` | 表格列名或列索引 |
| `cell_text` / `cell_text_contains` / `cell_text_regex` | 单元格文本匹配 |
| `tree_path` | 树节点路径数组 |
| `menu_path` | 菜单路径数组 |
| `scroll_to` | `start`、`end`、`visible`、`target` |
| `visible_only` | 是否只处理可见控件 |

### Backend 策略

Windows 当前主线应继续使用本地 Python backend，先用 UI Automation pattern 覆盖能力。Microsoft UI Automation 将控件能力抽象为 control patterns，客户端通过 pattern 读取或操作控件行为；当前 Windows 实现应优先利用 Grid、Table、Selection、SelectionItem、ExpandCollapse、Scroll、Invoke、Value 等 pattern。

macOS 侧保持 AX 抽象字段一致，后续在真机上用 AXRole、AXTitle、AXValue、AXIdentifier、AXChildren、AXSelectedRows、AXMenuItem 等能力映射。不能为了 Windows UIA 字段破坏跨平台契约。

### 输出 Payload

复杂控件 payload 必须保留原始控件证据和可压缩摘要：

```json
{
  "ok": true,
  "action": "desktop_element",
  "type": "get_table",
  "desktop": "desk",
  "window": {},
  "element": {},
  "table": {
    "row_count": 10,
    "column_count": 4,
    "columns": ["ID", "Name", "Status", "Amount"],
    "rows": []
  },
  "diagnostics": {
    "patterns": ["Grid", "Table", "Selection"],
    "visible_only": true,
    "truncated": false
  }
}
```

大表格和大树必须支持 `max_rows`、`max_nodes`、`text_limit`，避免把全量控件树塞进变量和 AI 上下文。完整产物写文件，变量里只放摘要和路径。

首批 `get_table/select_cell` 已采用的参数和 payload：

| 能力 | 参数 | Payload |
| --- | --- | --- |
| `get_table` | `max_rows`、`max_columns`、`visible_only`、`text_limit` | `table.row_count`、`table.column_count`、`table.columns`、`table.rows`、`table.cells`、`table.truncated` |
| `select_cell` | `row` + `column` 或 `column_index` | `selected_cell.row`、`selected_cell.column_index`、`selected_cell.text/value/name`、`selected_cell.bounds`、`table.columns` |

`get_table` 计入桌面识别/读取证据；`select_cell` 是操作推进步骤，不能单独替代读取、断言、截图或等待证据。`select_cell` 成功后写入 `desktop-annotations/` 标注。

## 窗口和控件相对视觉

### 目标

当前 `desktop_capture region`、`desktop_capture target=window/element`、`desktop_vision region` 和 `desktop_vision source_target=window/element` 已可用：

- 已支持截取指定窗口、指定控件。
- 已支持在窗口或控件内直接做 `locate_image`。
- 已支持视觉命中结果保留局部坐标和屏幕全局坐标。
- `desktop_input target=bounds_center` 默认消费全局坐标，避免点击错位。

### Action 扩展

`desktop_capture type=screenshot` 已支持 `target`：

```json
{
  "action": "desktop_capture",
  "desktop": "desk",
  "type": "screenshot",
  "target": "window",
  "title_contains": "Demo",
  "path": "demo-window.png",
  "save_as": "window_shot"
}
```

```json
{
  "action": "desktop_capture",
  "desktop": "desk",
  "type": "screenshot",
  "target": "element",
  "title_contains": "Demo",
  "automation_id": "OrdersGrid",
  "path": "orders-grid.png",
  "save_as": "grid_shot"
}
```

规则：

- `target=window` 必须提供 Window Query。
- `target=element` 必须提供 Window Query 和 Element Locator。
- `target=region` 必须提供 `{x,y,width,height}` region。
- `target=window/element` 不能同时使用 `region`。
- payload 包含 `target`、`source_bounds`、`coordinate_space`，窗口截图包含 `window`，控件截图包含 `locator` 和 `element`。

`desktop_vision type=locate_image` 已支持 `source_target`：

```json
{
  "action": "desktop_vision",
  "desktop": "desk",
  "type": "locate_image",
  "source_target": "window",
  "title_contains": "Demo",
  "template_path": "resources/save-icon.png",
  "path": "save-icon.json",
  "save_as": "save_icon"
}
```

规则：

- `source_target=screen` 抓取当前屏幕作为 source。
- `source_target=window` 必须提供 Window Query。
- `source_target=element` 必须提供 Window Query 和 Element Locator。
- `source_path` 不能和 `source_target` 同时使用。
- `region` 会限制 source 内搜索范围；`source_target=window/element` 时 region 是窗口或控件截图内的局部区域。
- `match_index` 只表示图像候选序号，不作为窗口序号；多窗口同名时使用更具体的 Window Query 或 `window_id`。

### 坐标契约

视觉和截图 payload 必须明确三类坐标：

```json
{
  "coordinate_space": {
    "origin": "screen",
    "unit": "logical_px",
    "scale": null
  },
  "source_bounds": {"x": 100, "y": 80, "width": 900, "height": 600},
  "coordinate_diagnostics": {
    "local_to_global_offset": {"x": 100, "y": 80},
    "scale": null,
    "warnings": ["scale_unknown"]
  },
  "match": {
    "bounds": {"x": 420, "y": 180, "width": 32, "height": 32},
    "local_bounds": {"x": 320, "y": 100, "width": 32, "height": 32}
  }
}
```

规则：

- `bounds` 永远是屏幕逻辑像素，可直接给 `desktop_input target=bounds_center`。
- `local_bounds` 相对于 `source_bounds`。
- `source_bounds` 来自窗口 bounds、控件 bounds 或显式 region。
- 多显示器和 DPI/Retina 信息放入 `coordinate_space`、`display` 和 `diagnostics`。

## OCR Locator

### 目标

OCR 是视觉兜底，不替代控件树。它适合：

- 自绘 UI 只有文字可见。
- 图片按钮旁边有稳定文本。
- 控件树缺失但截图可读。
- 需要读屏幕上的短文本或状态提示。

不适合：

- 大段文档抽取。
- 小字号、低对比度、动态视频、复杂表格识别。
- 高风险操作的唯一定位依据。

### Action

```json
{
  "action": "desktop_vision",
  "desktop": "desk",
  "type": "locate_text",
  "text_contains": "保存",
  "language": "chi_sim+eng",
  "min_confidence": 0.7,
  "region": {"x": 0, "y": 0, "width": 1200, "height": 800},
  "path": "save-text.json",
  "save_as": "save_text"
}
```

字段：

| 字段 | 作用 |
| --- | --- |
| `text` | 精确文本 |
| `text_contains` | 包含文本 |
| `text_regex` | 正则匹配文本 |
| `language` | OCR 语言，默认 `eng`，中英混排可用 `chi_sim+eng` |
| `min_confidence` | 最低置信度，取值 `0..1` |
| `source_path` | 已有截图 |
| `source_target` | `screen`、`window`、`element` |
| `region` | 限定区域 |
| `max_matches` | 最大候选 |
| `path` | 输出 JSON |

### Runtime 选择

第一阶段 OCR 建议用 Tesseract 作为跨平台可选能力，因为它可以命令行或 API 方式处理图片，并支持多语言。Windows.Media.Ocr 可作为后续 Windows 专属 provider，但官方文档说明该 namespace 对桌面应用要求 package identity，也就是 MSIX 打包身份；这不适合作为当前 Python 开发态的默认路径。

### Payload

```json
{
  "ok": true,
  "action": "desktop_vision",
  "type": "locate_text",
  "desktop": "desk",
  "provider": "tesseract",
  "language": "chi_sim+eng",
  "raw_text": "保存\n取消",
  "ocr_blocks": [],
  "matches": [
    {
      "index": 0,
      "text": "保存",
      "confidence": 0.91,
      "bounds": {"x": 420, "y": 180, "width": 48, "height": 22},
      "local_bounds": {"x": 20, "y": 10, "width": 48, "height": 22}
    }
  ],
  "match": {},
  "artifacts": {
    "json_path": "...",
    "source_path": "...",
    "annotation_path": "..."
  },
  "diagnostics": {
    "engine_version": "",
    "tessdata": "",
    "preprocess": []
  }
}
```

OCR 失败必须输出 source 图、raw OCR 文本、候选块、置信度和诊断 JSON，不能退化为猜坐标。

## 坐标校准

坐标校准服务应统一服务截图、控件 bounds、视觉定位和输入：

- 记录 displays：id、bounds、work_area、scale、primary。
- 记录截图来源：screen、window、element、region。
- 记录单位：logical_px、physical_px。
- 记录转换链：physical -> logical、local -> screen。
- 标注产物里保留所有参与点击的 point/bounds 和 source image。

建议新增内部 helper，不新增 action：

```python
CoordinateMapper.from_snapshot(snapshot)
CoordinateMapper.local_to_screen(local_bounds, source_bounds)
CoordinateMapper.screen_to_local(screen_bounds, source_bounds)
```

后续多显示器负坐标和 Retina 必须先进入 self-check，再开放给 AI 终端生成复杂视觉 plan。

## 能力矩阵

`capability_matrix` 应扩展但保持向后可读：

```json
{
  "capabilities": {
    "semantic": {
      "table": true,
      "tree": true,
      "menu": true,
      "scroll_container": true
    },
    "screenshot": {
      "window": true,
      "element": true,
      "region": true
    },
    "vision": {
      "image_locator": true,
      "ocr": true,
      "relative_source": true
    },
    "coordinates": {
      "multi_display": false,
      "dpi_aware": true,
      "retina_aware": false
    }
  },
  "dependencies": {
    "tesseract": true,
    "tessdata.eng": true,
    "tessdata.chi_sim": false
  }
}
```

AI 终端必须先看能力矩阵。缺表格 pattern 时不要写表格 action；缺 OCR runtime 时不要写 `locate_text`；坐标校准未知时不要把视觉命中直接用于高风险点击。

## Self-check

当前高级桌面能力已先并入 `desktop-components`：

```powershell
python .\cplan.py self-check desktop-components
python .\cplan.py self-check desktop-components --require-vision
python .\cplan.py self-check desktop-components --require-ocr
```

高级能力继续增多后可以再拆出独立入口，避免默认矩阵过重；拆分前以 `desktop-components` 为权威回归入口。

建议分层：

- `schema`: 新增 type 和字段负例。
- `fixtures`: Windows WinForms/WPF 临时表格、树、菜单、滚动容器；macOS 真机上使用 Tkinter/系统轻量 App 补验证。
- `relative_capture`: 窗口截图、控件截图、窗口内图像定位。
- `ocr`: 自生成图片 + 中英文短文本 OCR；缺依赖返回 `skipped`。
- `coordinate`: 多显示器/DPI 信息结构化检查；当前机器无法覆盖时返回 `skipped` 和原因。
- `ai`: `inspect_desktop` 返回扩展 capability matrix，`review_plan_quality` 按能力矩阵阻止不可用高级 action。

## 风险

- 复杂控件在 WPF、WinForms、Qt、Electron、自绘 UI 中暴露程度差异很大。
- 表格和树可能很大，必须限制输出数量并把全量证据写文件。
- 菜单和上下文菜单存在短生命周期窗口，必须有等待和失败截图。
- OCR 依赖本机二进制、语言包、图像质量和预处理；不能作为第一定位方式。
- 多显示器和 DPI/Retina 会导致截图像素与鼠标逻辑坐标不一致，必须先校准再开放高风险点击。
- macOS AX 字段弱于 Windows AutomationId，复杂控件需要真机补充映射和回归。
