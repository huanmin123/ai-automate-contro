# 跨平台电脑控制远期设计

## 定位

这份文档只记录远期设计，不作为当前开发范围。

当前阶段继续优先打磨浏览器自动化内核，包括 Playwright 会话、页面、定位、输入、网络、事件、截图、断言、文件读写、HTTP 客户端请求、受控同步命令、控制流和调试能力。电脑控制、系统信息、进程管理、桌面应用控制等能力暂不进入近期实现。

远期目标是让同一套 JSON plan 可以在受控边界内编排浏览器、系统命令、进程状态、桌面窗口和必要的键鼠输入，形成可复用、可调试、可持续运行的本地自动化流水线。

## 设计原则

- plan schema 保持稳定，平台差异收敛到 backend adapter。
- 优先使用语义化、可观测、可回放的控制方式，最后才使用坐标级键鼠输入。
- Windows 和 macOS 都必须是一级目标，不把 Windows 专用库设计成通用核心。
- 高风险能力必须有权限边界、超时、日志、输出限制和人工确认。
- 不让开放式 AI 直接执行系统命令或自由控制电脑；AI 只能辅助创建、检查、调试 plan。

## 能力分层

推荐按可靠性从高到低分层：

1. 应用自身 API、CLI、配置文件或导入导出接口。
2. 浏览器自动化，继续使用 Playwright。
3. 系统命令和脚本，已提供第一版受控同步 `command` action。
4. 系统信息和进程管理，统一封装为 `system` / `process` action。
5. 桌面窗口和控件级自动化，统一封装为 `desktop` action，并按平台选择 backend。
6. 截图、OCR、图像识别和坐标级键鼠输入，只作为兜底方案。

## 跨平台 backend 方向

### 系统信息与进程

候选核心依赖：

- `psutil`: 读取 CPU、内存、磁盘、网络、进程列表、进程状态等跨平台信息。

远期 action 草案：

```json
{
  "action": "system",
  "type": "snapshot",
  "include": ["cpu", "memory", "disk", "network"],
  "save_as": "system_status"
}
```

```json
{
  "action": "process",
  "type": "list",
  "name": "chrome",
  "save_as": "chrome_processes"
}
```

### 命令执行

第一版 `command` action 已落地，支持同步运行、任意本机工作目录、超时、输出限幅、stdout/stderr 落盘和 JSON stdout 解析。按本地调试原文优先策略，不做危险命令片段拦截。

命令执行使用 Python 标准库 `subprocess` 封装，不直接暴露裸执行器。

Windows 默认 backend 可以使用 PowerShell 7 (`pwsh`)；macOS 默认 backend 可以使用系统 shell。plan 应支持按平台提供不同命令，避免在一个字段里混合平台差异。

当前 action 示例：

```json
{
  "action": "command",
  "type": "run",
  "commands": {
    "windows": "Get-Process | Select-Object -First 5",
    "macos": "ps aux | head -n 5"
  },
  "timeout_ms": 30000,
  "save_as": "process_preview"
}
```

### 桌面窗口与控件

桌面自动化需要 adapter 层，不能依赖单一跨平台库覆盖全部能力。

候选 backend：

- Windows: `pywinauto`、`pywin32`、PowerShell、UI Automation。
- macOS: AppleScript/JXA、Accessibility API、Quartz、必要时使用 `pyobjc`。
- 通用兜底: `pyautogui`、`pynput`、`mss`、OpenCV、OCR。

`PyDirectInput` 仅适合作为 Windows 下的特殊兜底能力，主要用于普通输入库不生效的 DirectInput 场景。它不应进入跨平台核心依赖，也不应成为默认桌面控制方案。

远期 action 草案：

```json
{
  "action": "desktop",
  "type": "window",
  "operation": "focus",
  "title_contains": "Untitled"
}
```

```json
{
  "action": "desktop",
  "type": "input",
  "backend": "auto",
  "operation": "hotkey",
  "keys": ["ctrl", "s"]
}
```

```json
{
  "action": "desktop",
  "type": "capture",
  "path": "screen.png"
}
```

## 权限与本地执行边界

这些能力一旦实现，默认必须比浏览器 action 更严格：

- 命令执行必须支持超时、工作目录限制、输出大小限制和退出码记录。
- 危险命令、删除文件、修改系统配置、终止进程等操作不做关键词拦截；是否执行由用户和 plan 内容决定。
- 所有产物仍写入当前 plan 包 `output/`，不能写入源码目录、`resources/` 或仓库其他位置。
- macOS 需要提示用户授予 Accessibility、Screen Recording、Automation 等系统权限。
- Windows 需要处理 UAC、管理员权限、安全桌面和不同用户会话隔离。
- plan 校验阶段应能识别平台限定字段、危险字段和缺失的确认策略。

## 依赖拆分草案

远期可以用 optional dependencies 拆分：

```toml
[project.optional-dependencies]
system = ["psutil"]
desktop = ["pyautogui", "pynput", "mss"]
vision = ["opencv-python"]
windows-desktop = [
  "pywinauto; sys_platform == 'win32'",
  "pywin32; sys_platform == 'win32'",
  "pydirectinput; sys_platform == 'win32'"
]
macos-desktop = [
  "pyobjc-framework-Quartz; sys_platform == 'darwin'",
  "pyobjc-framework-ApplicationServices; sys_platform == 'darwin'"
]
```

## 暂缓项

当前不实现以下内容：

- `system`、`process`、`desktop` action。
- 后台进程、常驻服务和完整系统命令权限模型。
- Windows/macOS 桌面 backend。
- OCR、图像匹配、坐标级输入。
- 常驻调度、文件监听和长期后台运行。

这些能力等浏览器自动化主线稳定后，再单独开计划文档和测试包推进。
