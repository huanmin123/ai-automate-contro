# 打包发布指南

项目支持把 CLI 打成可分发目录，并把目录压成 zip，输出统一放到仓库根目录的 `out/`。当前只支持：

- Windows：在 Windows 机器上打 Windows 可执行文件。
- macOS：只支持 Apple Silicon，在 Apple Silicon macOS 上打 macOS 可执行文件。

不支持 Linux，也不支持在 Windows 上打 macOS 包。PyInstaller 不是跨平台交叉编译器；要发布多个平台，就必须分别在对应平台执行打包。

## 打包边界

打包产物包含 Python 运行时、项目代码、Python 依赖、Playwright Chromium、`handbook/` 和 `plan.config`。因为 Chromium 会一起打进去，文件会比较大，这是浏览器自动化工具的正常成本。

脚本会显式关闭 UPX 压缩，避免本机 PATH 上存在 UPX 时产生不可压缩 warning 或增加杀软误报概率。

打包脚本每次都会先清理旧的分发目录和 zip；如果旧分发包正在运行，脚本会先尝试结束相关进程，避免打出来的不是最新版。

清理动作只允许发生在仓库 `out/` 和系统临时 PyInstaller 工作目录下，避免路径拼写或同名前缀目录导致误删。Windows 和 macOS 脚本都会重新复制当前 Playwright Chromium 到发行目录的 `_internal/playwright/driver/package/.local-browsers`，避免旧浏览器文件或 PyInstaller 数据收集差异影响浏览器 smoke。

打包产物默认不复制 `test-plans/config.json`、`.keygen/` 会话状态或 `output/` 运行产物，也不会把旧 `out/ai-automate-contro/plans/config.json` 的本地配置恢复进新的分发目录；这是分发目录来源规则，不是因为真实密钥或敏感字段被拦截。用户运行 exe 时，默认以 exe 所在目录作为运行根目录；也可以通过 `AI_AUTOMATE_PROJECT_ROOT` 指定其他运行根。

发行目录结构：

```text
out/
  ai-automate-contro/
    aic.exe
    _internal/
    handbook/
    plan.config
    plans/
      config.json
  ai-automate-contro-windows-x64.zip
```

`plan.config` 控制 handbook 和 plan 集合位置：

```json
{
  "handbook_path": "handbook",
  "plan_roots": [
    "plans"
  ],
  "default_ai_config_dir": "plans"
}
```

这些路径默认相对于 exe 所在目录。需要真实 AI 服务时，在发行目录的 `plans/config.json` 里添加 `ai_services.default`，或把 `default_ai_config_dir` 指向你自己的配置目录。

发行包使用 PyInstaller onedir 模式：`_internal/` 存放 Python 运行时和依赖，启动时不需要每次解包，比单文件 exe 更适合这个项目。分发时请发送整个 `ai-automate-contro/` 目录或 zip，不要只复制 exe。

AI 终端文本搜索仍然依赖本机安装的 `rg`。目标机器如果要正常使用 AI 终端，需要先安装 `ripgrep`。

## Windows 打包

推荐使用专用打包虚拟环境，避免全局 Python 里无关包、旧 PyInstaller hook 或 PATH 上的工具影响打包结果。第一次准备环境时，在 PowerShell 7 中执行：

```powershell
Set-Location F:\python\ai-automate-contro
python -m venv .\.venv-package
.\.venv-package\Scripts\python.exe -m pip install -U pip
.\.venv-package\Scripts\python.exe -m pip install -e ".[package]"
```

日常正式打包优先使用这条命令：

```powershell
Set-Location F:\python\ai-automate-contro
.\scripts\package.ps1 -PythonPath ".\.venv-package\Scripts\python.exe" -NoProcessCleanup -SkipPlaywrightInstall
```

`-PythonPath` 固定使用专用虚拟环境；`-NoProcessCleanup` 避免脚本主动扫描并结束旧打包/分发进程；`-SkipPlaywrightInstall` 复用已经安装到 `.venv-package` 的 Playwright Chromium，减少重复下载和大量文件写入。首次机器上没有浏览器时，先单独安装：

```powershell
Set-Location F:\python\ai-automate-contro
$env:PLAYWRIGHT_BROWSERS_PATH = "0"
.\.venv-package\Scripts\python.exe -m playwright install chromium
Remove-Item Env:\PLAYWRIGHT_BROWSERS_PATH -ErrorAction SilentlyContinue
```

输出目录和 zip：

```text
out/ai-automate-contro/
out/ai-automate-contro-windows-x64.zip
```

主程序文件名固定为 `aic.exe`，便于启动；zip 文件名仍会跟随当前平台和机器架构变化，例如 `ai-automate-contro-windows-arm64.zip`。

PyInstaller 的工作目录放在系统临时目录，脚本结束后会清理；仓库根目录只保留 `out/` 里的最终产物。

### Windows 蓝屏或系统崩溃排查

打包脚本、Python 或 PyInstaller 是用户态程序，正常失败应该表现为 Python 异常、PyInstaller 报错或脚本退出码非 0；不应该直接导致 Windows 蓝屏。如果打包时出现 `CRITICAL_PROCESS_DIED`、机器重启或系统级崩溃，先按系统/驱动/硬件稳定性问题处理，不要反复直接跑完整打包。

2026-06-21 在 Windows 11 + Python 3.13.2 + PyInstaller 6.21.0 上做过一次分段诊断：标准库、Rich、Textual、Playwright Python 包、OpenAI/LangChain/LangGraph/Pillow 组合小项目都能独立打包并运行；Playwright Chromium 下载解压、主项目 PyInstaller 分析、浏览器目录复制、约 389 MB zip 压缩也都能通过。结论是：当前项目的 Python 依赖组合本身不是稳定蓝屏触发点。更可疑的是当时的全局 Python 环境、旧脚本进程清理、杀软/驱动/VSS/硬件状态或系统稳定性。

遇到类似情况时，先使用诊断输出后缀，不覆盖正式 `out\ai-automate-contro`：

```powershell
Set-Location F:\python\ai-automate-contro
.\scripts\package.ps1 -PythonPath ".\.venv-package\Scripts\python.exe" -OutputSuffix diag -ProbeOnly
.\scripts\package.ps1 -PythonPath ".\.venv-package\Scripts\python.exe" -OutputSuffix diag -NoProcessCleanup -SkipZip -StopAfter DependencyCheck
.\scripts\package.ps1 -PythonPath ".\.venv-package\Scripts\python.exe" -OutputSuffix diag -NoProcessCleanup -SkipZip -StopAfter PlaywrightInstall
.\scripts\package.ps1 -PythonPath ".\.venv-package\Scripts\python.exe" -OutputSuffix diag -NoProcessCleanup -SkipZip -SkipPlaywrightInstall -StopAfter PyInstaller
.\scripts\package.ps1 -PythonPath ".\.venv-package\Scripts\python.exe" -OutputSuffix diag -NoProcessCleanup -SkipZip -SkipPlaywrightInstall -SkipBrowserCopy -StopAfter CopyPackage
.\scripts\package.ps1 -PythonPath ".\.venv-package\Scripts\python.exe" -OutputSuffix diag -NoProcessCleanup -SkipZip -SkipPlaywrightInstall -StopAfter CopyPackage
.\scripts\package.ps1 -PythonPath ".\.venv-package\Scripts\python.exe" -OutputSuffix diag -NoProcessCleanup -SkipZip -SkipPlaywrightInstall
.\scripts\package.ps1 -PythonPath ".\.venv-package\Scripts\python.exe" -OutputSuffix diag -NoProcessCleanup -SkipPlaywrightInstall
```

每一步都会把检查点写到 `out\package-last-diag.log`。如果某一步导致重启，重启后先看这个日志最后一行，判断停在 `PlaywrightInstall`、`Run PyInstaller`、`Copy Playwright browsers into package` 还是 `Compress package zip`。不要跳过阶段直接重试完整打包。

诊断包生成后，先跑离线自检：

```powershell
Set-Location F:\python\ai-automate-contro
.\out\ai-automate-contro-diag\aic.exe self-check env
.\out\ai-automate-contro-diag\cplan.exe self-check cli
.\out\ai-automate-contro-diag\cplan.exe self-check runtime
.\out\ai-automate-contro-diag\cplan.exe validate --file .\out\ai-automate-contro-diag\plans\demo\plan.json
```

`out\package-last.log` 或 `out\package-last-diag.log` 是判断打包卡点的第一证据；Windows 事件查看器、`C:\Windows\Minidump\*.dmp` 和 WHEA 事件用于判断系统层问题。磁盘盘符或 D 盘格式化本身通常不会直接影响打包，除非 `TEMP`、pip cache、PyInstaller cache、项目路径或 Playwright 浏览器目录实际指向了该盘。

## macOS Apple Silicon 打包

macOS 使用原生 shell 脚本打包，不需要 PowerShell 7。脚本默认使用 PATH 里的 `python3`，也可以通过 `--python` 或 `PYTHON` 指定解释器；它不会默认创建项目 `.venv`。

```bash
cd /path/to/ai-automate-contro
./scripts/package-macos.sh --install-dependencies --clean --smoke-test
```

输出文件：

```text
out/ai-automate-contro/
out/ai-automate-contro-macos-arm64.zip
```

分发包包含两个入口：`aic` 是 AI-first 客户端，`cplan` 是无 AI plan 控制 CLI。

如果当前 macOS 不是 Apple Silicon，脚本会直接拒绝打包。

## 使用打包产物

更完整的用户侧启动、配置、AI 终端、图片粘贴和调试流程见 [分发包使用指南](./分发包使用指南.md)。这里保留最小 smoke 命令，方便打包后快速确认产物可用。

建议从发行目录运行，方便真实调试：

```powershell
Set-Location F:\python\ai-automate-contro\out\ai-automate-contro
.\cplan.exe self-check runtime
.\cplan.exe create --path .\plans\demo
.\cplan.exe validate --file .\plans\demo\plan.json
.\cplan.exe run --file .\plans\demo\plan.json --run-name packaged-smoke
.\aic.exe ai --thread packaged-demo
```

也可以显式指定工作区：

```powershell
$env:AI_AUTOMATE_PROJECT_ROOT = "F:\python\ai-automate-contro"
.\out\ai-automate-contro\cplan.exe self-check runtime
```

`self-check runtime` 只检查 `plan.config`、`handbook/` 和 plan 集合目录。配置好真实 AI 服务后，再运行 `self-check env` 检查 AI 配置结构。

AI 终端创建 plan 时，如果用户没有明确给目录，会默认使用 `plan.config.plan_roots[0]`。这个发行包默认就是 `plans/`，所以新 plan 会落到发行目录的 `plans/<name>/`。

如果把 exe 发给别人，对方至少需要：

- 一个包含 `plan.json` 的工作区。
- PowerShell 7，用于按文档执行命令。
- `ripgrep`，用于 AI 终端渐进式文本搜索。
- 可访问的 OpenAI-compatible 模型服务配置，如果要使用 AI 终端或专项 AI action。

## 发布前检查

打包前建议先跑源码侧验证：

```powershell
python -m py_compile .\main.py (Get-ChildItem .\src -Recurse -Filter *.py | ForEach-Object { $_.FullName })
python .\main.py self-check env
python .\main.py self-check ai-terminal
python .\main.py self-check ai-tools
python .\main.py tool check
python .\cplan.py self-check cli
python .\cplan.py self-check runtime
python .\cplan.py self-check browser-components
python .\cplan.py validate --file .\plans\minimal-browser-plan\plan.json
```

打包后建议用 exe 再跑：

```powershell
Set-Location .\out\ai-automate-contro
.\aic.exe self-check env
.\aic.exe self-check ai-stream
.\cplan.exe self-check runtime
.\aic.exe self-check ai-terminal
.\aic.exe self-check textual-client
.\aic.exe self-check ai-tools
.\aic.exe tool check
.\cplan.exe validate --file .\plans\demo\plan.json
.\cplan.exe run --file .\plans\demo\plan.json --run-name packaged-smoke
```

`out/` 是本地打包产物目录，不应提交到 Git。
